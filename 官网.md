对，**Redis 不能丢**。但目标不是“去 Redis”，而是把 Redis 从 **核心控制面** 降级成 **辅助加速层**。

你现在的问题是：

```text
Redis = queue + cron lock + cache + pubsub + throttler + config snapshot + 幂等
```

应该改成：

```text
Cloud Tasks / Jobs = queue + cron
Redis = cache + 限流 + 幂等 + 短期状态
Postgres = 关键状态 / 最终事实
```

## Redis 仍然适合保留的能力

这些可以继续用 Redis：

```text
1. 业务缓存
   food detail cache
   recommendation cache
   scoring config snapshot
   user profile read cache

2. 限流
   API rate limit
   AI 分析频率限制
   登录/邮件发送频率限制

3. 幂等短 key
   webhook event id
   payment event id
   task dedupe key

4. 短期状态
   analysis progress
   temporary upload/session state
   recent decision cache

5. 分布式锁
   少量、低频、非关键锁可以留
```

## Redis 不应该继续承担的能力

这些最好逐步拿掉：

```text
1. BullMQ 主队列
2. 高频 cron lock
3. 关键任务状态
4. 长链路 workflow 状态
5. 多模块强依赖的 Pub/Sub 控制面
```

## 关键原则

### 1. Redis 挂了，核心业务不能挂

例如：

```text
推荐缓存 Redis 挂了
→ 可以回源 DB / 实时计算

限流 Redis 挂了
→ 可以 fail-open 或本地内存兜底

幂等 Redis 挂了
→ 关键支付/webhook 必须查 Postgres 幂等表
```

### 2. 关键状态必须进 Postgres

这些不要只放 Redis：

```text
analysis status
subscription event
payment webhook event
task execution result
recommendation precompute result
decision record
```

Redis 只做加速。

### 3. Redis 需要拆职责

最少拆成：

```env
CACHE_REDIS_URL=xxx
QUEUE_REDIS_URL=xxx   # 如果短期还保留 BullMQ
```

如果迁移到 Cloud Tasks / Jobs 后：

```env
CACHE_REDIS_URL=xxx
```

就够了。

## 最终你的架构会变成

```text
Cloud Run API
  ├─ Postgres：核心数据 / 状态 / 幂等 / 任务结果
  ├─ Redis：缓存 / 限流 / 短 TTL / 快速幂等
  ├─ Cloud Tasks：用户触发异步任务
  └─ Cloud Run Jobs + Scheduler：批量/定时任务
```

## 所以答案

**不是丢 Redis。**

而是：

```text
Redis 继续保留
但不要再承担 queue / cron / 系统控制面
```

这样 Redis 就算出问题，系统最多是：

```text
变慢、少部分功能降级
```

而不是：

```text
API 启动失败、worker 崩、任务丢、缓存失效全挂
```
