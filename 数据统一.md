你现在是一个系统架构师 + NestJS专家 + 数据规范治理专家。

当前系统存在严重的字段不统一问题：

- 数据库：snake_case（下划线）
- 后端API：camelCase + 部分嵌套
- 前端：admin / web 各自处理字段（不统一）

---

# 🎯 你的任务

设计一套【统一字段标准方案】，并给出可执行的改造方案：

目标是：

1. API DTO 成为唯一数据标准
2. 前端不再做字段转换
3. AI输出直接符合 API 结构
4. 消除所有多余 mapping

---

# 🚫 强约束（必须遵守）

- 不改变数据库原有字段命名（允许继续 snake_case）
- 不破坏现有业务逻辑
- 所有改造必须“渐进式可上线”
- 不允许一次性重构整个系统

---

# Step 1：现状问题分析（必须）

分析：

- DB → API 存在什么转换
- API → 前端 是否存在二次转换
- AI输出是否统一

输出：
- 字段混乱点
- 风险点（如重复字段）

---

# Step 2：统一标准定义（核心）

定义：

## 唯一标准：

- API 输出统一使用 camelCase
- 所有字段以 DTO 为准

---

## 数据流规范：

DB（snake_case）
→ Entity（camelCase）
→ DTO（camelCase，最终输出）
→ 前端（直接使用）

---

# Step 3：后端改造方案（NestJS）

必须设计：

---

## （1）DTO规范（核心）
- 所有接口必须返回 DTO
- DTO字段统一 camelCase

---

## （2）字段映射策略
- DB字段 → Entity自动映射
- Entity → DTO 统一转换

---

## （3）统一序列化
- 使用 class-transformer
- 全局拦截器控制输出

---

## （4）禁止行为
- 不允许 Controller 直接返回 Entity
- 不允许返回 snake_case

---

# Step 4：前端改造方案（admin + web）

要求：

- 完全依赖 API 字段
- 删除所有字段转换逻辑

输出：

- 如何清理 mapping
- 如何统一类型定义

---

# Step 5：AI补全适配（关键）

必须设计：

- AI输出必须符合 DTO结构
- 不允许自由字段

输出：

- AI prompt如何约束字段
- 如何做结构校验

---

# Step 6：数据统一层（建议实现）

设计一个：

👉 Data Mapper Layer

作用：

- 所有数据进入系统前统一转换
- AI / 第三方 / DB 全部统一

---

# Step 7：渐进式迁移方案（非常重要）

分阶段：

Phase 1：
- 新接口统一 DTO

Phase 2：
- 老接口逐步替换

Phase 3：
- 前端去除所有 mapping

---

# Step 8：风险控制

必须说明：

- 如何避免字段冲突
- 如何避免数据错乱
- 如何回滚
