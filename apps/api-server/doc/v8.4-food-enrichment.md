## 二、V8.4 变更详情

### 2.1 Rollback camelCase Bug 修复（Phase 1）

**问题**：`rollbackEnrichment()` 中构建 `rollbackUpdates` 时直接用 `enrichedFields`（snake_case）的 key 传入 `tx.foods.update({ data: ...rollbackUpdates })`，Prisma 要求 camelCase key，导致字段回退静默失败（Prisma 忽略未知字段）。

**修复**：新增 `rollbackUpdatesCamel`，所有字段通过 `snakeToCamel()` 转换后再写入 Prisma。`mergedFood` 完整度预计算也改用 camelCase 的合并对象。

```typescript
// 修复前（错误）
const rollbackUpdates: Record<string, any> = {};
for (const field of enrichedFields) {
  rollbackUpdates[field] = null; // snake_case key → Prisma 忽略
}
await tx.foods.update({ data: { ...rollbackUpdates } }); // ← Bug

// 修复后（正确）
const rollbackUpdatesCamel: Record<string, any> = {};
for (const field of enrichedFields) {
  rollbackUpdatesCamel[snakeToCamel(field)] = null; // camelCase key ✓
}
const mergedFood = { ...food, ...rollbackUpdatesCamel }; // 完整度预计算也用 camelCase
await tx.foods.update({ data: { ...rollbackUpdatesCamel } }); // ✓
```

### 2.2 Rollback 改为删除原日志（无审计日志）（Phase 1）

**背景**：需求明确"回退只清除补全字段，不需要审计日志"。

**修复前**：

- 写一条 `ai_enrichment_rollback` 新日志
- 将原日志的 action 改为 `ai_enrichment_rolled_back`
- 共 2 次 change_log 写入

**修复后**：

- 直接 `DELETE` 原日志（`tx.foodChangeLogs.delete({ where: { id: logId } })`）
- 不写任何新日志
- 事务内只有 1 次 foods.update + 1 次 logs.delete

```typescript
await this.prisma.$transaction(async (tx) => {
  await tx.foods.update({ where: { id: log.foodId }, data: { ...rollbackUpdatesCamel, ... } });
  await tx.foodChangeLogs.delete({ where: { id: logId } }); // 删除原记录，不留审计
});
```

### 2.3 5阶段→1条汇总 Change Log（Phase 1）

**背景**：`enrichFoodNow()` 原先在循环里每个阶段调一次 `applyEnrichment()`，每次写一条 `food_change_logs`，5个阶段最多产生 5 条日志，查询和回滚复杂。

**修复**：将所有阶段结果合并为一个 `EnrichmentResult`，只调一次 `applyEnrichment()` / `stageEnrichment()`，只写一条汇总日志。

**合并策略**：

- 字段值：先来先得，后阶段不覆盖前阶段已填充的字段
- 置信度：使用 `multiResult.overallConfidence`（各阶段加权平均）
- fieldConfidence：先到先得合并
- reasoning：各阶段 reasoning 用 `|` 连接
- staged 判定：任意阶段置信度低于阈值（0.7）→ 整体走 staged 模式

```typescript
// 合并所有阶段结果
const mergedFields: Record<string, any> = {};
for (const sr of multiResult.stages) {
  for (const [k, v] of Object.entries(sr.result)) {
    if (v !== null && v !== undefined && !(k in mergedFields)) {
      mergedFields[k] = v; // 先来先得
    }
  }
}
// 只调一次，只写一条日志
await this.applyEnrichment(foodId, mergedResult, 'ai_enrichment_now');
```

### 2.4 /enrichment/stats Redis 降级保护（Phase 1）

**问题**：BullMQ 队列统计（`getWaitingCount()` 等）依赖 Redis。Redis 不可用时整个 `/stats` 接口抛 500，连 DB 侧历史统计也无法返回。

**修复**：用 `safeQueueStat` 包装每个队列调用，异常时降级返回 0，不影响 `getEnrichmentHistoricalStats()` 的正常返回。

```typescript
const safeQueueStat = async (fn: () => Promise<number>): Promise<number> => {
  try {
    return await fn();
  } catch {
    return 0;
  }
};
```

## 三、核心架构

### 3.1 补全流程（单条立即补全）

```
POST /enrichment/enrich-now/:foodId
        │
        ▼
enrichFoodNow(foodId, options)
        │
        ├─ 自动检测缺失字段 → 确定 targetStages
        │
        ▼
enrichFoodByStage(foodId, targetStages)
        │
        ├─ Stage 1: 核心营养素 (protein/fat/carbs/fiber/sugar/sodium)
        ├─ Stage 2: 微量营养素 (23个字段)
        ├─ Stage 3: 健康属性  (glycemic_index/fodmap/allergens等)
        ├─ Stage 4: 使用属性  (meal_types/cuisine/common_portions等)
        └─ Stage 5: 扩展属性  (ingredient_list/dish_type/prep_time等)
                │
                ▼
        合并所有阶段结果为一个 EnrichmentResult
                │
        ┌───────┴────────┐
        │ confidence≥0.7 │ confidence<0.7
        ▼                ▼
  applyEnrichment()  stageEnrichment()
  (直接入库)          (暂存待审)
        │                │
        └───────┬─────────┘
                ▼
        1条 food_change_logs 汇总记录
```

### 3.2 Change Log 动作类型

| action                          | 触发时机                     | 日志条数                     |
| ------------------------------- | ---------------------------- | ---------------------------- |
| `ai_enrichment`                 | `applyEnrichment()` 直接入库 | 1条/次调用                   |
| `ai_enrichment_staged`          | `stageEnrichment()` 暂存     | 1条/次                       |
| `ai_enrichment_approved`        | `approveStaged()` 审核通过   | 原staged日志更新             |
| `ai_enrichment_rejected`        | `rejectStaged()` 审核拒绝    | 原staged日志更新             |
| ~~`ai_enrichment_rollback`~~    | ~~rollback~~                 | **V8.4 已删除，直接 DELETE** |
| ~~`ai_enrichment_rolled_back`~~ | ~~rollback~~                 | **V8.4 已删除**              |

### 3.3 字段名转换规则

```
ENRICHABLE_FIELDS (snake_case) ←→ Prisma model (camelCase)
        glycemic_index              glycemicIndex
        food_group                  foodGroup
        meal_types                  mealTypes
        ...

转换工具：snakeToCamel(s) = s.replace(/_([a-z])/g, (_, c) => c.toUpperCase())

使用场景：
  - applyEnrichment()    prismaUpdates[snakeToCamel(k)] = v   ✓
  - rollbackEnrichment() rollbackUpdatesCamel[snakeToCamel(f)] = null  ✓ (V8.4修复)
  - computeCompletenessScore() food[snakeToCamel(field)]        ✓
```

## 四、关键 API 接口

### 4.1 单条立即补全

```
POST /admin/food-pipeline/enrichment/enrich-now/:foodId
Body: { stages?: number[], fields?: string[], staged?: boolean }
```

### 4.2 补全队列统计（含 Redis 降级）

```
GET /admin/food-pipeline/enrichment/stats
返回: { queue: { waiting, active, completed, failed, delayed }, historical: { ... } }
注: Redis 不可用时 queue 全返回 0，historical 仍正常
```

### 4.3 全库补全进度

```
GET /admin/food-pipeline/enrichment/progress
返回: { totalFoods, fullyEnriched, partiallyEnriched, notEnriched, avgCompleteness, stagesCoverage, byStatus }
注: 首次使用需先调用 POST /recalculate-completeness 回填历史数据
```

### 4.4 批量重算完整度（历史数据回填）

```
POST /admin/food-pipeline/enrichment/recalculate-completeness
Body: { batchSize?: number }  // 默认 200
功能: 对全库食物重算 data_completeness 和 enrichment_status
注: 进度面板为空时运行一次即可
```

### 4.5 回退补全

```
POST /admin/food-pipeline/enrichment/rollback/:logId
功能: 将补全字段重置为 null，删除原 change_log 记录（V8.4: 不写审计日志）
```

### 4.6 聚合轮询（Phase 2）

```
GET /admin/food-pipeline/enrichment/dashboard-poll
返回: {
  queue:      { waiting, active, completed, failed, delayed }  // BullMQ，Redis 不可用时降级为 0
  historical: { total, enriched, pending, failed, staged, avgCompleteness }
  recentLogs: [ { id, foodId, foodName, action, enrichedFields, confidence, createdAt }, ... ]  // 最近10条
  pendingReview: number         // 当前待审核 staged 数
  avgCompleteness: number
  byStatus:   { pending, enriched, staged, failed, ... }
}
用途: 前端单接口轮询，实时感知入队/处理/审核全流程数据流转
```

### 4.7 历史日志字段级对比（Phase 2）

```
GET /admin/food-pipeline/enrichment/history/:logId/diff
限制: 仅支持 action=ai_enrichment / ai_enrichment_approved 类型
返回: {
  logId, foodId, foodName, action, operator, createdAt, confidence,
  diff: [{ field, enrichedValue, currentValue, isCurrent, fieldConfidence }],
  enrichedFields, reasoning
}
注: currentValue 为 foods 表实时值，isCurrent=false 表示补全后已被后续操作覆盖
```

### 4.8 批量进度快照（Phase 2）

```
GET /admin/food-pipeline/enrichment/batch-status
返回: {
  queue:      { waiting, active, completed, failed, delayed }
  processed:  { total, enriched, pending, failed, staged, avgCompleteness }
  remaining:  waiting + active（队列中尚未处理的估算数）
  completionRate: number（%，基于 DB 数据）
}
用途: 批量入队后轮询此接口，追踪整体批次进度
```

### 4.9 审核细粒度报表（Phase 3）

```
GET /admin/food-pipeline/enrichment/review-stats
返回: {
  pendingReview:          number     // 当前待审核数
  approved:               number     // 历史审核通过总数
  rejected:               number     // 历史审核拒绝总数
  reviewed:               number     // approved + rejected
  approvalRate:           number     // 通过率 %（小数2位）
  rejectionRate:          number     // 拒绝率 %（小数2位）
  avgConfidenceAll:       number     // 所有已审核记录的平均置信度
  avgConfidenceApproved:  number     // 已通过记录的平均置信度
  avgConfidenceRejected:  number     // 已拒绝记录的平均置信度
  confidenceBuckets: [               // 置信度区间分布（5个桶）
    { bucket: '0.0-0.2', approved, rejected },
    { bucket: '0.2-0.4', approved, rejected },
    { bucket: '0.4-0.6', approved, rejected },
    { bucket: '0.6-0.8', approved, rejected },
    { bucket: '0.8-1.0', approved, rejected },
  ]
  dailyTrend: [{ date, approved, rejected }]  // 最近30天按日趋势
  pendingList: [{ logId, foodId, foodName, enrichedFields, confidence, createdAt }]  // 积压概要（最多20条）
}
用途: 运营侧感知审核效率、置信度分布、积压情况，独立于 operations-stats 的细粒度视图
```

## 五、Phase 3 修复详情

### 5.1 fieldConfidence 解析 Bug 修复

**问题**：`callAIForStage()` 的 prompt 要求 AI 返回 `"field_confidence"` (snake_case)，但代码只解析 `raw.fieldConfidence` (camelCase)，导致字段级置信度永远为空。

**修复**：兼容两种格式：

```typescript
const rawFieldConf = raw.field_confidence ?? raw.fieldConfidence ?? {};
// 字段查找也兼容 snake_case / camelCase 两种 key
const fc = rawFieldConf[field] ?? rawFieldConf[snakeToCamel(field)] ?? null;
```

### 5.2 NUTRIENT_RANGES key 不匹配修复

**问题**：`NUTRIENT_RANGES` 用 camelCase key（如 `glycemicIndex`），但 `validateAndClean()` 传入的 `field` 是 snake_case（如 `glycemic_index`），导致所有数值范围校验全部失效（静默通过超出范围的数值）。

**修复**：查找时同时尝试两种格式：

```typescript
const range = NUTRIENT_RANGES[field] ?? NUTRIENT_RANGES[snakeToCamel(field)];
```

### 5.3 AI Prompt 增强

**system prompt 新增内容**：

- 5个权威数据来源列表（USDA FoodData Central、中国食物成分表等）
- "推算值也必须给出，不能返回 null"原则
- 置信度分级说明：
  - ≥0.85：有权威来源
  - 0.6-0.85：合理推算
  - 0.4-0.6：粗略估计
  - <0.4：基本猜测

**user prompt 变更**：

- 旧：`"无法确定的字段返回 null"`
- 新：`"推算值也必须给出，仅对完全无法评估的字段才返回 null"`

## 六、运维说明

1. **首次部署后**：调用 `POST /recalculate-completeness` 回填历史数据，进度面板方可正常显示
2. **Redis 不可用**：`/stats`、`/dashboard-poll`、`/batch-status` 接口队列部分返回 0，不影响 DB 历史统计
3. **Rollback 行为变更**：V8.4 起 rollback 直接删除 change_log，不可撤销，操作前请确认
4. **前端轮询建议**：入队后轮询 `GET /batch-status` 追踪进度；需要实时日志和状态分布用 `GET /dashboard-poll`；审核运营视图用 `GET /review-stats`
