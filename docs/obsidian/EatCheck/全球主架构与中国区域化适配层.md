# 全球主架构与中国区域化适配层

## 结论

EatCheck 现在最正确的路线是：

> 全球主系统 + 中国区域化适配层

不是：

> 中国版 / 海外版双系统

当前不要急着做 ICP、国内支付、中国 AI、双服务器、双数据库。现在应该优先让海外全球版跑通，同时提前补齐 Provider Abstraction 和 Region Strategy。

## 当前项目基础

当前仓库已经具备区域化适配的基础：

- NestJS + Prisma + PostgreSQL
- AI Gateway 已存在，并已有 OpenAI、OpenRouter、DeepSeek、Qwen adapter
- Provider/model 管理已存在
- `LlmService` 已是统一 LLM 调用入口，但仍需 routed 模式
- `regionCode`、`timezone`、`locale` 已进入推荐链路
- Auth 已有 Firebase、手机号、微信、小程序、邮箱
- Billing 已有 RevenueCat、微信支付、支付记录和订阅权益系统

## 架构原则

### 核心业务统一

以下能力不拆区域版本：

- 推荐系统
- 饮食决策系统
- AI Prompt 与解释逻辑
- 用户画像与行为画像
- 食物数据库与 Prisma schema 主体
- 订阅权益语义
- 营养计算、评分、推荐过滤、反馈闭环

原因：这些是 EatCheck 的产品壁垒。双系统会让 AI 产品质量、数据口径、实验结果和研发节奏分裂。

### 基础设施区域化

以下能力允许按区域替换：

- Auth
- AI
- Billing
- Storage
- Push
- SMS
- Moderation
- Deployment

## 目标架构

```text
Global Flutter App / Web / Miniapp
        |
        v
API Gateway / BFF
        |
        v
Region Strategy Resolver
        |
        +--------------------+--------------------+
        |                    |                    |
   Global Runtime       China Runtime        Future Runtime
        |                    |                    |
 Firebase/OpenAI      SMS/WeChat/Qwen       EU-specific stack
 RevenueCat/GCP       WeChat/Alipay/OSS     Data residency layer
        |
        v
Unified Core Domain
Recommendation / Profile / Food DB / Subscription Entitlements / Analytics
```

## Phase 1 边界

Phase 1 只做中国兼容架构，不做中国正式上线。

不做：

- ICP
- 中国正式支付
- 中国 AI 正式生产切换
- 双服务器
- 双数据库
- 双 App

要做：

- [[Provider Abstraction 抽象清单]]
- [[Region Strategy 落地方案]]
- `/api/app/capabilities`
- AI Gateway routed mode
- Auth/Billing provider registry
- provider/model 的 region 配置
