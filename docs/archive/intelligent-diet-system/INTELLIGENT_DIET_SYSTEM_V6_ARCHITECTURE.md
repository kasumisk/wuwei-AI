# 智能饮食推荐系统 V6 — 架构升级设计

> **版本**: V6.0 设计稿  
> **日期**: 2026-04-10  
> **定位**: 从"智能推荐系统"升级为"可持续进化的 AI 饮食推荐与商业化系统"  
> **原则**: 渐进演进，不推翻 V5，优先低成本高收益

---

## 1. V6 总体目标

**一句话**: V5 是"一个会推荐食物的 AI 系统"，V6 是"一个能自我进化、可商业化运营、并具备持续优化能力的 AI 饮食系统"。

V5 完成了推荐引擎的智能化基建（10 维评分、协同过滤、在线学习、pgvector、A/B 测试）。但系统仍然是：

- **被动的** — 用户不来就不推荐，没有主动触达
- **单体的** — 模块间直接注入，无事件驱动，无异步队列
- **不可运营的** — 没有订阅体系，没有功能分层，没有商业闭环
- **演进缓慢的** — 中文硬编码，策略与解释强耦合，新增能力需要频繁改代码

V6 的本质升级是三个"从…到…"：

| 维度     | V5             | V6                               |
| -------- | -------------- | -------------------------------- |
| AI 能力  | 单次请求式推荐 | 持续学习 + 多策略融合 + 主动推送 |
| 系统架构 | 同步直连单体   | 事件驱动 + 异步队列 + 分层缓存   |
| 商业化   | 零商业能力     | 订阅分层 + 功能门控 + 转化优化   |

### 3.2 各模块详细设计

#### 3.2.1 EventBusModule — 域事件总线

**问题**: V5 所有模块通过 `@Inject()` 直接引用服务，形成紧耦合。例如 `FeedbackService` 提交反馈后需要手动调用 `WeightLearnerService`、更新用户画像、记录 A/B 数据 — 每个新需求都要改 Feedback 代码。

**方案**: 引入 `@nestjs/event-emitter`，定义域事件（Domain Events），让各模块自行订阅。

```
核心事件定义:
  user.feedback.submitted    → 触发: 权重学习 / 画像更新 / A/B 数据记录
  user.meal.recorded         → 触发: 行为画像更新 / 连续性检测 / 成就检查
  user.profile.updated       → 触发: 缓存失效 / 推荐预计算 / 细分重算
  recommendation.generated   → 触发: 预缓存 / 解释生成 / 用量计数
  subscription.changed       → 触发: 功能门控刷新 / 缓存清除
  user.goal.achieved         → 触发: 成就解锁 / 推送通知 / 教练消息
```

**技术选型**: `@nestjs/event-emitter`（进程内），不引入 Kafka/RabbitMQ（单体架构够用，短期内无需引入更重的分布式消息系统）。

**改造成本**: 低。定义事件类 → 在关键操作点 emit → 各模块 `@OnEvent` 订阅。不改变现有业务逻辑，只是解耦调用方式。

#### 3.2.2 QueueModule — 异步任务队列

**问题**: V5 所有计算都是同步的。协同过滤矩阵重建、权重学习、画像批量更新都在 Cron 中同步执行，阻塞 Node 事件循环。AI 图片分析（GPT-4V）耗时 3-10 秒，直接阻塞请求。

**方案**: 引入 BullMQ（已有 Redis 基础设施）。

```
队列定义:
  profile-update        并发=5, 优先级=中    画像实时更新
  recommendation-precompute  并发=3, 优先级=低    每日推荐预计算
  feedback-process      并发=10, 优先级=高   反馈处理（权重学习 + 画像更新）
  food-analysis         并发=3, 优先级=中    AI 图片分析（长耗时任务）
  notification          并发=10, 优先级=中   推送/站内信发送
  export                并发=2, 优先级=低    数据导出
```

**技术选型**: `@nestjs/bullmq` + Redis（复用现有 Redis 7 实例，生产环境建议单独 Redis 实例）。

**改造成本**: 中。需安装 bullmq，定义 Queue + Processor，将现有 Cron 任务迁移为 Queue Job。但不影响业务逻辑本身。

#### 3.2.3 StrategyEngineModule — 推荐策略引擎

**问题**: V5 推荐逻辑硬编码在 `recommendation-engine.service.ts`（864 行）。权重、餐次角色、优化维度都是常量。想要"新用户多探索、老用户多个性化"需要改代码。A/B 实验只能覆盖权重数值，不能切换整个策略。

**方案**: 抽象出策略层，将推荐流程拆解为可编排的策略单元。

```
策略引擎核心概念:

  Strategy          推荐策略（完整的推荐方案）
  ├── RecallPolicy      召回策略（向量/CF/规则/热门 的选择与权重）
  ├── RankPolicy        排序策略（评分维度权重/评分公式）
  ├── RerankPolicy      重排策略（探索率/多样性/时间衰减）
  ├── OptimizePolicy    优化策略（全局优化维度/迭代轮数）
  └── ContextPolicy     上下文策略（画像注入/场景检测）

  StrategyResolver    根据用户画像 + A/B 实验 → 决定使用哪个 Strategy
```

**存储**: `strategy` 表存策略配置 JSON，`strategy_assignment` 表存用户→策略映射。支持 A/B 实验直接分配不同策略（而不仅仅是覆盖权重）。

**改造成本**: 中高。需要将 `recommendation-engine.service.ts` 的硬编码逻辑抽象为策略接口，但 Pipeline 三阶段架构（Recall → Rank → Rerank）本身不变，只是每阶段的参数变为可配置。

#### 3.2.4 FeatureFlagModule — 功能开关

**问题**: V5 没有灰度发布能力。新功能只能全量上线或不上线。A/B 实验系统只覆盖推荐权重，无法控制功能可见性。

**方案**: 轻量级功能开关系统（不引入 LaunchDarkly 等重量级方案）。

```
功能开关类型:
  BOOLEAN       开/关（全局或按用户）
  PERCENTAGE    百分比放量（如 10% 用户可见新功能）
  USER_LIST     白名单/黑名单
  SEGMENT       按用户画像段开放（如只对 muscle_builder 开放）

存储: feature_flag 表 + Redis 缓存（30s TTL）
```

**与 A/B 实验的关系**: 功能开关决定"是否可见"，A/B 实验决定"用哪个版本"。两者配合：先用 Feature Flag 灰度放量，再用 A/B 测试优化效果。

**改造成本**: 低。独立模块，不影响现有逻辑。在需要灰度的代码点加 `if (featureFlag.isEnabled('xxx', userId))` 判断。

#### 3.2.5 RealtimeProfileModule — 实时画像管道

**问题**: V5 画像更新依赖 Cron（日/周/双周），用户行为到画像反映有 24 小时延迟。用户连续拒绝 3 个推荐，系统要到第二天才"知道"。

**方案**: 事件驱动的实时画像更新 + 短期/长期画像分离。

```
画像分层:

  短期画像 (ShortTermProfile)
  ├── 存储: Redis Hash（7 天滑窗）
  ├── 更新: 每次用户行为事件即时更新
  ├── 内容: 近 7 天口味偏好 / 拒绝模式 / 活跃时段 / 摄入趋势
  └── 用途: 推荐时实时注入上下文

  长期画像 (LongTermProfile)
  ├── 存储: PostgreSQL（现有 UserInferredProfile）
  ├── 更新: Cron 批量（保留现有机制）
  ├── 内容: 稳定偏好 / 段落标签 / 健康趋势 / 营养缺口
  └── 用途: 策略选择 / 细分 / 长期趋势分析

  合并策略: 短期权重 0.6 + 长期权重 0.4（可配置）
  回溯: 画像变更日志表 (profile_change_log) — 记录变更原因 + 前后值 + 触发事件
```

**技术实现**: 监听 `user.meal.recorded` / `user.feedback.submitted` 等事件 → 通过 BullMQ `profile-update` 队列 → 更新 Redis 短期画像。不阻塞请求主线程。

**改造成本**: 中。新建短期画像服务 + Redis 结构 + 事件监听。现有长期画像逻辑不变。

#### 3.2.7 PrecomputeModule — 离线预计算

**问题**: V5 所有推荐都是实时计算。对于固定场景（每天三餐），完全可以提前计算好。用户打开 App 时 200ms 返回预计算结果，而不是等 500ms+ 实时计算。

**方案**: 基于 BullMQ 的离线预计算管道。

```
预计算策略:

  每日推荐预计算（凌晨 3:00）
  ├── 触发: Cron → 为所有活跃用户（7 天内有行为）生成次日三餐推荐
  ├── 存储: precomputed_recommendation 表（用户/日期/餐次/结果 JSON）
  ├── 命中: 用户请求推荐时先查预计算表
  ├── 回退: 预计算未命中（新用户/画像变更/手动请求）→ 实时计算
  └── 失效: 用户画像变更事件 → 删除该用户当日预计算

  用户活跃时段预热（早 7:00 / 中 11:00 / 晚 17:00）
  ├── 触发: Cron → 预热即将活跃用户的缓存（画像/食物池）
  └── 效果: 首次请求延迟从 500ms → 200ms

数据模型:
  precomputed_recommendation
  ├── userId, date, mealType
  ├── result (JSONB — MealRecommendation)
  ├── strategyVersion (策略版本，用于失效判断)
  ├── computedAt, expiresAt
  └── isUsed (用户是否实际使用了此预计算)
```

**改造成本**: 中。主要是调用现有推荐引擎 + 存储结果 + 请求时优先读取。核心推荐逻辑零改动。

#### 3.2.8 ExplainV2Module — 可视化 + 交互式解释

**问题**: V5 的 `ExplanationGeneratorService` 输出纯文案。前端只能显示文字。没有结构化可视化数据，没有反向解释。

**方案**: 升级解释输出格式，支持可视化和交互。

```
解释输出结构 V2:

  ExplanationV2
  ├── summary: string                    一句话解释（现有）
  ├── primaryReason: string              推荐主因（现有）
  ├── healthTip: string                  健康提示（现有）
  ├── radarChart: RadarChartData         雷达图数据（10 维评分可视化）
  │   └── { dimensions: [{ name, score, weight, benchmark }] }
  ├── progressBars: ProgressBarData[]    进度条数据（营养素 vs 目标）
  │   └── [{ nutrient, current, target, unit, status }]
  ├── comparisonCard: ComparisonData     对比卡片（与用户平均/健康标准对比）
  │   └── { vsUserAvg, vsHealthyTarget, trend7d }
  ├── whyNotExplanation?: string         反向解释（"为什么没推荐 X？"）
  ├── upgradeTeaser?: string             付费预览（"升级查看完整分析"）
  └── locale: string                     语言标识

反向解释 API:
  POST /app/food/explain-why-not
  Body: { foodName: string, mealType: string }
  Response: { reason: string, alternatives: ScoredFood[] }
```

**改造成本**: 低-中。在现有 `ExplanationGeneratorService` 基础上扩展输出字段。核心评分逻辑已有，只需转换为可视化数据结构。

#### 3.2.9 I18nModule — 国际化框架

**问题**: V5 所有用户面文案中文硬编码（解释文案、教练回复、成就描述、错误消息）。`i18n-messages.ts` 只有 36 条中文。V5 设计文档明确说"不引入 i18next"。

**V6 策略**: 不引入重量级 i18n 框架，采用**渐进式国际化** — 先做"可翻译"架构，不做"全翻译"。

```
国际化分层:

  L1 — 系统消息 i18n（Phase 2 实现）
  ├── 错误消息、验证消息、通知消息
  ├── 实现: 扩展现有 i18n-messages.ts → 按 locale 分文件
  ├── 新增 locale: en-US, ja-JP（按市场优先级）
  └── 不改变现有接口，只是 t() 函数支持 locale 参数

  L2 — 推荐解释 i18n（Phase 2 实现）
  ├── 解释模板参数化（现在是直接拼中文字符串）
  ├── 模板: "推荐 {{food}}，因为其 {{nutrient}} 含量 {{level}}"
  └── 每个 locale 一套模板

  L3 — 食物数据 i18n（中长期预研）
  ├── 复用现有 food_translation 表
  ├── 扩展: 食物名 / 食物描述 / 标签 多语言
  └── 推荐结果中的食物名按用户 locale 返回

  L4 — 营养标准 i18n（中长期预研）
  ├── DRI 参考值按地区可配置（中国/美国/日本/WHO）
  ├── 单位系统可切换（公制/英制）
  └── 存储: nutrition_standard 配置表
```

**改造成本**: L1/L2 低，L3/L4 中。关键是改造 `ExplanationGeneratorService` 和 `CoachService` 为模板化输出。

#### 3.2.10 NotificationModule — 主动触达

**问题**: V5 系统完全被动 — 用户不打开 App 就无任何交互。已安装 `firebase-admin` 但未使用。

**方案**: 推送 + 站内信。

```
推送场景:
  meal_reminder         餐次提醒（"午餐时间到了，查看今日推荐"）
  streak_risk           连续性风险（"还差 1 次记录就能保持 7 天连续"）
  goal_progress         目标进展（"本周已达成 5/7 天目标"）
  weekly_report         周报就绪（"您的周度营养报告已生成"）
  coach_nudge           教练提醒（"距上次聊天已 3 天"）
  precomputed_ready     推荐就绪（"今日餐单已备好"）

技术:
  iOS: APNs（通过 firebase-admin）
  Android: FCM（通过 firebase-admin）
  站内信: notification 表 + 未读计数 API
  偏好: 用户可配置接收哪些通知、免打扰时段
```

**改造成本**: 中。firebase-admin 已安装，需要实现推送服务 + 通知偏好 + 站内信 CRUD。

## 4. 架构升级重点

### 4.1 必须重构的部分

| 重构项                | 原因                                                                               | 影响范围                         | 预估工作量 |
| --------------------- | ---------------------------------------------------------------------------------- | -------------------------------- | ---------- |
| **引入 EventEmitter** | 模块间直接注入导致紧耦合，每加一个监听方都要改触发方代码                           | 全局（但每个模块改动极小）       | 2-3 天     |
| **引入 BullMQ**       | 无异步能力，Cron 同步阻塞，AI 调用阻塞请求                                         | diet / user / food-pipeline 模块 | 3-4 天     |
| **推荐引擎策略抽象**  | `recommendation-engine.service.ts` 864 行单文件，硬编码策略参数，无法运行时切换    | recommendation 子模块            | 5-7 天     |
| **解释系统输出格式**  | 当前纯文案输出无法支撑可视化 / i18n / 商业化                                       | explanation-generator            | 2-3 天     |
| **缓存层统一**        | 各服务手写缓存逻辑（FoodPoolCache / ProfileCache / AB 缓存），模式重复但实现不一致 | 各缓存服务                       | 3-4 天     |

### 4.2 可增量实现的部分

| 增量项          | 说明                                           | 前置依赖    | 预估工作量      |
| --------------- | ---------------------------------------------- | ----------- | --------------- |
| **FeatureFlag** | 独立模块，按需在代码点加判断                   | 无          | 2 天            |
| **短期画像**    | 新增 Redis 结构 + 事件监听，不影响现有长期画像 | EventBus    | 3-4 天          |
| **预计算管道**  | 调用现有推荐引擎 + 存储结果                    | BullMQ      | 3-4 天          |
| **订阅模块**    | 独立模块 + 装饰器/守卫，不侵入业务逻辑         | FeatureFlag | 7-10 天         |
| **通知模块**    | firebase-admin 已安装，独立模块                | EventBus    | 3-4 天          |
| **i18n 框架**   | 渐进替换硬编码字符串                           | 无          | 5-7 天（L1+L2） |
| **多目标优化**  | 扩展现有全局优化器，不替换                     | 策略引擎    | 5-7 天          |

### 4.3 架构演进路径

```
V5 架构（当前）:

  Client → Controller → Service → Service → Repository → PostgreSQL
                                     ↓
                                   Redis（手写缓存）
                                     ↓
                                   Cron（同步批处理）

V6 架构（目标）:

  Client → Controller → Guard(订阅/限流) → Service → EventBus ──→ 监听者们
               ↓                              ↓                    ├── 实时画像更新
          FeatureFlag                    策略引擎                   ├── 权重学习
               ↓                    ┌─────┴──────┐                ├── 成就检查
          功能门控                 规则策略    模型策略              ├── 通知推送
                                     ↓                            └── 运营触达
                                  BullMQ 队列
                              ┌──────┴──────────┐
                           同步（推荐请求）   异步（预计算/画像/分析）
                              ↓                    ↓
                        三级缓存             Worker 处理
                    L1 内存 → L2 Redis → L3 预计算表 → PostgreSQL
```

**关键原则**: 不微服务化。V6 仍然是 NestJS 单体，但内部采用事件驱动 + 队列解耦，短期聚焦推荐、商业化和工程效率提升，不引入平台化范围。

## 5. 技术风险 & 复杂度评估

### 5.1 高风险 / 高复杂度

| 风险项             | 复杂度 | 风险说明                                                       | 缓解策略                                                                          |
| ------------------ | ------ | -------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| **支付集成**       | ★★★★★  | Apple IAP 审核严格，微信支付对接复杂，退款/掉单处理繁琐        | 先上 Apple IAP（iOS 强制），微信支付延后；使用成熟 SDK；充分的异常处理 + 对账机制 |
| **策略引擎抽象**   | ★★★★☆  | 过度抽象会导致性能下降和调试困难；抽象不够又无法灵活配置       | 先抽象 Rank 策略（最高收益），Recall/Rerank 保持硬编码；用 JSON 配置而非 DSL      |
| **实时画像一致性** | ★★★☆☆  | 短期画像和长期画像可能冲突（短期说用户爱甜，长期说用户在减糖） | 定义明确的合并策略；短期画像设 TTL 自动过期；合并时长期画像有最低权重保底         |

### 5.2 中风险

| 风险项            | 复杂度 | 风险说明                                 | 缓解策略                                                   |
| ----------------- | ------ | ---------------------------------------- | ---------------------------------------------------------- |
| **BullMQ 可靠性** | ★★★☆☆  | 队列任务丢失或重复执行                   | 幂等设计 + 死信队列 + 监控告警；Redis 持久化（AOF）        |
| **预计算时效性**  | ★★★☆☆  | 用户画像变了但预计算未更新，推荐结果过时 | 画像变更事件 → 立即失效预计算；请求时校验策略版本          |
| **缓存一致性**    | ★★★☆☆  | 三级缓存带来更多一致性问题               | 统一缓存抽象层；缓存版本号机制；宁可多 miss 不可返回脏数据 |
| **i18n 覆盖度**   | ★★☆☆☆  | 遗漏硬编码中文字符串                     | CI 检查：正则扫描中文字符；渐进替换，不要求一次全覆盖      |

### 5.3 可快速落地（低风险高收益）

| 项目            | 复杂度 | 收益                                       | 落地时间 |
| --------------- | ------ | ------------------------------------------ | -------- |
| **EventBus**    | ★★☆☆☆  | 立即解耦模块通信，为后续所有异步能力打基础 | 2-3 天   |
| **FeatureFlag** | ★★☆☆☆  | 灰度发布能力，降低新功能上线风险           | 2 天     |
| **通知推送**    | ★★☆☆☆  | 用户触达能力，提升留存率                   | 3-4 天   |
| **解释可视化**  | ★★☆☆☆  | 产品差异化，提升用户感知价值               | 2-3 天   |
| **预计算**      | ★★☆☆☆  | P50 延迟从 500ms → 200ms，用户体验显著提升 | 3-4 天   |

| 支付安全 | 无       | 支付签名验证 + 对账 + 防重放 |
| API 安全 | 全局限流 | + 分层限流 + 请求签名校验    |

## 10. 关键决策记录

| 决策                                   | 选择             | 理由                                                                                                                                                            |
| -------------------------------------- | ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **不微服务化**                         | 保持 NestJS 单体 | 团队规模不支撑微服务运维成本；单体 + 事件驱动 + 队列已够用；短期优先把单体内部解耦和可运营能力做好                                                              |
| **BullMQ 而非 Kafka**                  | BullMQ           | 已有 Redis 基础设施；单体架构不需要分布式消息；BullMQ 支持优先级/延迟/重试/死信                                                                                 |
| **自建 FeatureFlag 而非 LaunchDarkly** | 自建             | 需求简单（布尔/百分比/白名单）；不想增加外部依赖；成本控制                                                                                                      |
| **渐进 i18n 而非 nestjs-i18n**         | 自建扩展         | 现有 i18n-messages.ts 模式简单够用；按 locale 分文件 + 模板化即可；避免过度框架化                                                                               |
| **Apple IAP 优先**                     | Apple IAP 先行   | iOS 强制要求；中国 iOS 用户付费习惯好；微信支付 Android 端后续跟进                                                                                              |
| **策略引擎用 JSON 配置而非 DSL**       | JSON             | DSL 学习成本高、维护成本高；JSON 可直接存 PostgreSQL JSONB，配合管理后台 UI 编辑                                                                                |
| **不引入外部 ML 框架**                 | 保持规则 + 统计  | V5 的规则引擎（10 维评分 + 在线学习 + Thompson Sampling）已经足够智能；引入 TensorFlow/PyTorch 需要 Python 服务，增加架构复杂度；当数据量达到百万级用户时再考虑 |

## 12. 总结

V6 不是推翻 V5 重来，而是在 V5 扎实的推荐引擎基础上，解决三个核心问题：

1. **系统不能自我进化** → 事件驱动 + 实时画像 + 策略引擎 + 预计算
2. **系统不能赚钱** → 订阅体系 + 功能门控 + 转化优化
3. **系统演进效率不足** → 国际化框架 + 可视化解释 + 统一缓存 + 灰度能力

每个 Phase 都有独立的可交付价值：

- **Phase 1（4-5 周）**: 性能提升 + 实时画像 + 推送 → 用户体验质变
- **Phase 2（6-8 周）**: 策略引擎 + 商业化 + 国际化 → 开始产生收入

短期 V6 完成预估 **10-13 周**（一个人全职）或 **6-8 周**（2-3 人团队）。
