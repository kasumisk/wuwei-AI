## 三、7 日推荐示例（代表性用户）

### 3.1 用户 CN-01（小李 / 减脂 / 低预算 / 不会做饭）

|------|------|------|------|---------|
| Day 1 | 豆浆(甜)+全麦面包 | **西红柿炒鸡蛋+米饭** | 凉拌木耳+麻婆豆腐+米饭 | ⚠️ 麻婆豆腐高油高盐 |
| Day 2 | 豆浆(甜)+燕麦片 | **西红柿炒鸡蛋+米饭** | 炒青菜+家常豆腐+米饭 | ❌ 午餐与Day1完全重复 |
| Day 3 | 酸豆奶+全麦面包 | **西红柿炒鸡蛋+拍黄瓜+米饭** | 凉拌木耳+豆腐+米饭 | ❌ 西红柿炒鸡蛋连续第3次 |
| Day 4 | 豆浆(甜)+燕麦 | **西红柿炒鸡蛋+炒豆芽** | 家常豆腐+炒青菜+米饭 | ❌ 西红柿炒鸡蛋连续第4次 |
| Day 5 | 全麦面包+牛奶 | 炒饭(蛋炒饭)+凉菜 | 炒藕片+家常豆腐+米饭 | ⚠️ 蛋炒饭=高碳水 |

### 4.2 各维度说明与评分依据

#### 合理性（68/100）

| 正向证据 | 负向证据 |
|---------|---------|
| 14维评分系统覆盖热量/蛋白/碳水/脂肪/血糖 | 召回层无 region 过滤，CN 食物进入 JP 用户结果 |
| 糖尿病/痛风/肾病有硬过滤规则 | SceneResolver 时区 bug：服务器时区判断用户餐次 |
| NOVA 加工食品惩罚逻辑存在 | inferred profile 不感知 regionCode，冷启动误推 |
| 餐次权重修正（早/午/晚/加餐）已实现 | 素食/乳糖不耐 hard filter 依赖多字段组合，存在漏网风险 |

**扣分原因**：对于 JP 和 AU 用户，推荐结果中会混入与其生活场景完全无关的食物（如中国菜肴），导致合理性分数在非 CN 地区显著下降。

#### 可执行性（58/100）— 最大问题维度

| 用户类型 | 推荐内容 | 可执行性评估 |
|---------|---------|------------|
| CN-01（不会做饭，外卖） | 西红柿炒鸡蛋、麻婆豆腐、凉拌木耳 | ❌ 这些是"做菜"而非"外卖可点" |
| JP-04（便利店为主） | 罐头黑豆、烹饪类中餐 | ❌ 便利店买不到，完全不现实 |
| US-04（低预算快餐文化） | Beans canned 系列 | ❌ 技术上便宜，但用户文化习惯不匹配 |
| US-06（学生，微波炉） | 需要烹饪的食物 | ❌ canCook=false 但仍推需要技能的食物 |
| AU-04（低预算） | 高质量食物降权但仍出现 | ⚠️ 价格软评分不足以彻底过滤 |

**根本原因**：
1. `budgetFilterEnabled` 在 `homeCook` 场景为 `false`（见 scene.types.ts:282）
2. `PriceFitFactor` 是**软评分（乘数 0.6-1.05）而非硬过滤**，超预算食物仍会出现在结果中
3. `canCook=false` + `takeoutFrequency=often` 没有联动 channel 强制约束
4. `budgetPerMeal` 在 UserProfiles schema 已存在但**全代码库 0 次读取精确货币值**（仅 estimatedCostLevel 1-5 粗粒度）

---

#### 多样性（46/100）— 最严重问题

**稳定性报告直接印证**：

```
推荐一致性 same-input meal: 0.067
（意味着：同一用户 60 次请求中，93.3% 的推荐彼此不同 ← 看起来好）

但：
Top stable meal signatures:
- 4x 西红柿炒鸡蛋 > 拍黄瓜 > 麻婆豆腐 > Beans, black...
- 4x 西红柿炒鸡蛋 > 炒豆芽 > 豆浆(甜) > 牛肉干 > 酸豆奶
- 3x 西红柿炒鸡蛋 > 凉拌木耳 > 麻婆豆腐 > ...
- 3x 西红柿炒鸡蛋 > 炒青菜 > 豆浆(甜) > ...

= "西红柿炒鸡蛋"在 60 次请求中出现 14 次（23.3%）
```

**系统一致性 vs 用户体验差异**：

| 指标 | 系统视角（技术） | 用户视角（体验） |
|------|----------------|----------------|
| 每次请求结果 | 93.3% 不同（看起来有多样性） | 同一食物（西红柿炒鸡蛋）每 4-5 次出现 1 次 |
| 7天 daily plan | diversity=1.0（每天不同组合） | 但核心食物被重复，用户感知"总是那几道菜" |
| 真正的多样性 | 食物级别 diversity score | 成分/烹饪方式级别的感知多样性 |

**根本原因**：
1. `duplicateHitRate = 0.66`（daily plan）— 每周有 66% 的日期在用同样的食物签名
2. 多样性惩罚逻辑存在（`weekExcludeNames` 跨日排重），但**种类池不够大**，CN 区域中高质量评分食物集中在少数菜品
3. exploration 率 `0.02-0.15`，StrategyAutoTuner 在 100 次交互后收敛到 `0.02`，几乎不探索新食物

---

#### 个性化（55/100）

| 个性化信号 | 是否真正生效 |
|-----------|------------|
| 用户目标（fat_loss/muscle_gain等） | ✅ 权重矩阵差异化，对评分有显著影响 |
| 饮食限制（素食/过敏原） | ✅ 硬过滤实现，基本可信 |
| 历史反馈（FactorLearner） | ⚠️ 仅 preference-signal/price-fit/regional-boost 3 维，且 passiveActivated=false |
| 地区偏好（cuisinePreferences） | ⚠️ 有评分加成但召回层不感知 |
| 预算 | ❌ 精确预算 0 次读取，仅粗粒度 1-5 等级 |
| 烹饪能力（canCook） | ⚠️ 影响 skill 过滤但不影响 channel 强制 |
| 生活方式（外卖频次、备餐意愿） | ❌ 基本未影响推荐池选择 |
| LLM 推断画像 | ❌ inferred profile 不感知 regionCode，冷启动偏差大 |

**问题**：两个画像相似（均为CN/减脂/中预算）但一个是"外卖族"、另一个是"备餐族"的用户，会得到**几乎相同的推荐结果**。

## 五、问题深度分析

### P1 — 食物池 Region 污染（最高优先级）

**位置**：`food-pool-cache.service.ts` 召回层

```
缓存键 = pool:{category}  ← 全球用户共享同一池
不含 region 维度

后果：
- US/JP/AU 用户的候选池中混入 CN 菜肴
- 靠打分层软降权，但 top-N 截断仍可能让污染食物进结果
- JP 便利店用户推出"西红柿炒鸡蛋"
```

**用户影响**：日本/澳洲用户打开 App，看到的是中国菜单。他们的第一反应是"这不是给我用的"，直接卸载。

---

### P2 — 多样性崩溃（极高优先级）

**直接证据**（来自稳定性报告）：
```
daily-plan regenerate x50:
- consistency: 0.02 (看起来好)
- duplicateHitRate: 0.66  ← 66% 重复！
```

**根因**：
1. CN 高评分食物高度集中在"西红柿炒鸡蛋"系列（高分、高 commonality、低 acquisitionDifficulty）
2. StrategyAutoTuner exploration 收敛到最低值 `0.02`，几乎不探索新食物
3. `weekExcludeNames` 跨日排重，但**排除的是食物名，而非食材/烹饪方式**

**用户影响**：3-5 天后用户感知"天天一样"，停止使用推荐功能。

---

### P3 — 可执行性与现实脱节

**代码证据**：
```typescript
// scene.types.ts:282
homeCook: {
  budgetFilterEnabled: false,  // ← 自煮场景关闭预算过滤！
}
```

**结合稳定性报告 fallback 触发率 0.2（20%）**：每 5 次推荐中有 1 次触发 fallback 兜底，说明正常推荐链在某些场景下质量不稳定。

| 不现实场景 | 具体表现 |
|-----------|---------|
| CN-01（不会做饭）拿到需要烹饪的菜 | 外卖平台根本点不到"西红柿炒鸡蛋" |
| JP-04（便利店）拿到罐头黑豆推荐 | 日本便利店没有此商品 |
| US-04（快餐文化）拿到健康罐头推荐 | 技术正确但用户打死不吃 |
| 预算 $8 用户拿到"高端"推荐降了 40% | 软降权后仍可能出现在结果里 |

### P4 — SceneResolver 时区 Bug

**位置**：`scene-resolver.service.ts:156, 276, 352`

```typescript
const dayOfWeek = new Date().getDay();   // 服务器时区（北京时间）
const hour = new Date().getHours();      // 服务器时区（北京时间）
```

**影响**：
- 美国东部用户（UTC-5）晚餐时间（18:00 EST = 07:00 CST），系统判断为"早餐时段"
- 早餐场景权重（glycemic x1.3, satiety x1.2）被错误应用到晚餐推荐
- 澳大利亚用户（UTC+11）深夜被误判为"午餐时段"

### P5 — 精确预算信号断链

**代码证据**：
```
UserProfiles.budgetPerMeal (Decimal) — schema 已存在
UserProfiles.currencyCode — schema 已存在

但：PriceFitFactor.isApplicable() 中：
  const hasExact = typeof declared?.['budgetPerMeal'] === 'number' ...

  → ProfileAggregator 是否将此字段传入 ctx？ 答案：文档指出 0 次读取
```

**影响**：所有用户的价格匹配都退化到路径 B（粗粒度 1-5 等级），导致：

- 美元/人民币/日元/澳元用户使用同一套价格等级，无货币区分

### P6 — 推荐"过于理想化"问题

**典型表现**：
- 给 discipline=low 的用户（经常暴食、意志力弱）推荐需要精确备餐的食物
- 给家庭用户（familySize>1）推荐只适合单人份的食物
- 给工作繁忙用户推荐需要 30 分钟以上烹饪的食物

**评分系统关注点**：executability 维度存在，但 `habit` goal 的 executability 权重仅 0.09（9%），实际影响有限。


### 判断依据

系统的技术架构是优秀的（14维评分、学习系统、健康规则引擎），但在"真实用户会不会用"这个终极问题上存在三个 showstopper：

**Showstopper 1：非 CN 用户拿到 CN 菜单**
- 一个日本用户打开 App，看到推荐是"西红柿炒鸡蛋"和"麻婆豆腐"
- 他不需要 App Review，他直接卸载
- 这会摧毁所有非中文市场的用户留存

**Showstopper 2：多样性崩溃 —— "天天一样"**
- duplicateHitRate = 0.66（66% 的日计划在用重复组合）
- 稳定性报告已经实测：西红柿炒鸡蛋在连续60次请求中持续出现
- 用户愿意给推荐系统 3-5 天的试用期；如果每天都看到一样的东西，7日留存为 0

**Showstopper 3：推荐"听起来健康但不现实"**
- 不会做饭的用户拿到做饭菜单
- 便利店依赖的用户拿到需要烹饪的食物
- 低预算用户的价格过滤是软的（还是会看到超预算食物）
- 这是"理想健康顾问"而非"真实生活伴侣"的行为模式

### 如果 CN 市场单独上线？

如果仅面向 CN 市场，且用户画像为"会做饭 + 中等预算 + 有健身意识"，则：
- CN 子集综合评分 64 分
- 多样性 49 分仍不合格（天天西红柿炒鸡蛋）
- 可执行性对外卖族仍是问题
- **结论：CN 限量内测可以，全量开放仍需修复多样性**
