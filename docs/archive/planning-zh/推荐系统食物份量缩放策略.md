
你是高级推荐系统架构师 + 营养算法工程师 + 后端 NestJS/Prisma 工程师。

## 背景

当前项目是 AI 饮食健康建议 App：EatCheck。

推荐引擎目前已经支持：
- 根据用户画像、地区、时区、季节、饮食偏好推荐食物
- 根据营养目标对食物份量进行缩放
- 生成每餐推荐组合

但现在存在一个重要问题：

当前推荐引擎为了贴合营养目标，会把很多食物都缩放成较小份量。  
这在算法上可能合理，但在真实饮食场景中不合理。

例如：
- 一些主菜、完整菜品、套餐、饮品、包装食品，不适合随意缩成 0.3 份、0.5 份
- 一些食物更适合“推荐少几个”，而不是“每个都缩小”
- 一些配菜、坚果、水果、谷物、调味品，则可以适度缩放
- 有些食物只能按固定单位推荐，比如 1 个鸡蛋、1 盒酸奶、1 瓶饮料、1 份三明治

我希望新增一套“基于食物类别/形态/属性的份量缩放规则”，避免所有食物都被机械缩放。

## 二、设计 Portion Scaling Policy

请设计并实现一套“份量缩放策略系统”。

目标：

不同类型食物使用不同的份量处理方式。

至少支持以下策略类型：

### 1. `scalable`

适合自由缩放的食物。

例如：
- 米饭
- 燕麦
- 面条
- 肉类原料
- 蔬菜
- 水果切块
- 坚果
- 豆类
- 酸奶散装
- 沙拉基础食材

规则：
- 可以按 gram 缩放
- 但需要设置 minServingG / maxServingG
- 不能缩到不合理的小份
- 推荐结果需要显示合理的份量说明

---

### 2. `limited_scalable`

可以小范围缩放，但不应该大幅缩放。

例如：
- 一份炒菜
- 一份沙拉
- 一份汤
- 一份主菜
- 一份便当中的单个菜品

规则：
- 允许在 0.75x ~ 1.25x 或类似范围内调整
- 不允许缩成 0.3 份、0.5 份
- 超出目标时优先减少食物数量，而不是继续缩小
- 输出时尽量显示为“约 1 份”或“半份/小份/正常份/大份”

---

### 3. `fixed_unit`

不适合缩放，只能按固定单位推荐。

例如：
- 鸡蛋
- 酸奶杯
- 牛奶盒
- 饮料瓶
- 蛋白棒
- 三明治
- 汉堡
- 包装零食
- 包装食品
- 一整份外卖/快餐单品

规则：
- 不允许生成 0.3 个、0.4 盒、0.6 瓶
- 只能推荐 1 个、2 个，或不推荐
- 如果营养目标超出，应该移除或替换，而不是缩放
- 输出必须显示完整单位

---

### 4. `not_scalable`

完全不适合缩放的食物。

例如：
- 套餐
- 组合餐
- 餐厅固定菜品
- 外卖套餐
- 预包装完整餐
- 高度标准化商品

规则：
- 不参与份量缩放
- 只能按默认标准份推荐
- 如果不符合营养目标，应该降低排序、替换或减少推荐数量
- 不应该生成奇怪份量

---

### 5. `condiment_or_micro`

调味品或微量食物。

例如：
- 酱油
- 橄榄油
- 沙拉酱
- 糖
- 蜂蜜
- 调味料

规则：
- 可以按小克重缩放
- 但必须有严格上限
- 不应该作为主要推荐食物
- 主要用于营养估算或搭配建议，不应该占据推荐主菜位

---

## 三、结合现有字段自动判断策略

请不要强依赖人工维护单一字段。

请基于现有 Food 字段设计策略推断逻辑，例如：

- `foodForm`
  - ingredient → 通常 scalable
  - dish → limited_scalable 或 not_scalable
  - semi_prepared → fixed_unit / limited_scalable

- `dishType`
  - combo_meal / set_meal → not_scalable
  - soup / salad / main_dish → limited_scalable
  - snack / drink → fixed_unit 或 limited_scalable

- `category / subCategory / group`
  - condiment / sauce / oil → condiment_or_micro
  - beverage → fixed_unit 或 limited_scalable
  - packaged_food → fixed_unit
  - staple / grain / vegetable / fruit / meat → scalable 或 limited_scalable

- `standardServingG`
  - 极小份量要谨慎
  - 极大份量要避免被缩到离谱

- `commonPortions`
  - 如果存在 piece / cup / bottle / pack / serving 等单位，优先用于 fixed_unit 判断

- `availableChannels`
  - restaurant / delivery / convenience 可能更偏固定份
  - home_cook 更适合 gram 缩放

- `tags / ingredientList`
  - 可辅助判断套餐、包装、饮品、调味品等

请实现一个独立的策略推断模块，例如：

- `PortionScalingPolicyResolver`
- `portion-scaling-policy.service.ts`
- `portion-scaling-policy.types.ts`

要求：
- 规则清晰
- 可单元测试
- 可配置
- 不把复杂逻辑散落在推荐引擎各处

## 四、推荐组合逻辑也要调整

不要只限制单个食物的缩放。

还需要调整“一餐推荐组合”的生成策略：

1. 如果一餐推荐食物数量过多，优先减少数量，而不是把每个食物都缩小
2. 每餐建议设置合理的食物数量范围，例如：
   - 早餐：2–4 个
   - 午餐：3–5 个
   - 晚餐：3–5 个
   - 加餐：1–2 个
3. 每餐需要有角色结构：
   - staple / 主食
   - protein / 蛋白
   - vegetable / 蔬菜
   - optional side / 可选配菜
   - drink / optional
4. 如果总营养超标：
   - 优先移除 optional side
   - 再替换高热量食物
   - 再对 scalable 食物做缩放
   - 最后才对 limited_scalable 小范围调整
   - fixed_unit / not_scalable 不应强行缩放
5. 不要为了贴合目标生成“每个食物都很小”的组合
6. 推荐结果应该更像真实一餐，而不是数学凑数

---

## 五、营养目标匹配逻辑调整

请检查当前营养目标匹配是否过度依赖份量缩放。

优化方向：

1. 目标匹配应分层：
   - 先选合理食物组合
   - 再做少量份量调整
   - 再做替换/删减
2. 不要把份量缩放作为唯一优化手段
3. 支持不同策略的惩罚分：
   - scalable 缩放惩罚低
   - limited_scalable 大幅缩放惩罚高
   - fixed_unit 缩放直接禁止
   - not_scalable 缩放直接禁止
4. 推荐排序应考虑：
   - 营养匹配
   - 饮食真实性
   - 用户可执行性
   - 地区/季节/偏好
   - 类别均衡
   - 份量合理性
