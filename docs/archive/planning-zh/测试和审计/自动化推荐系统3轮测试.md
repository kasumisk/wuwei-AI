你现在是一个【高并发系统工程师 + 推荐系统稳定性专家 + 分布式系统调试专家】。

⚠️ 当前系统已经完成两轮调试（bug修复 + 对抗测试）。

你的任务是：

# 对推荐系统进行“稳定性验证 + 压测 + 学习系统行为验证”

---

# 一、本轮目标

不是找单点 bug，而是验证：

1. 系统是否稳定
2. 是否存在隐性问题（抖动 / 漂移 / 数据污染）
3. 学习系统是否正常工作（不是失控）

---

# 二、必须执行的测试类型

## 1️⃣ 连续调用稳定性测试

对同一用户执行：

* 连续调用 50 ~ 100 次：

  * recommendMeal
  * recommendDailyPlan

必须记录：

* 推荐结果是否稳定
* 是否出现明显漂移
* 是否出现重复/异常食物
* explanation 是否一致

---

## 2️⃣ 并发测试（关键）

模拟：

* 20 / 50 / 100 并发用户
* 不同用户画像
* 不同 region / channel / goal

检查：

* 是否有 race condition
* 是否 cache 污染（用户A影响用户B）
* 是否出现推荐异常
* 是否有错误日志

---

## 3️⃣ Cache 行为验证

必须验证：

* Redis key 是否正确隔离（userId / region / goal）
* 是否出现旧数据污染
* cache miss / hit 是否合理
* cache 更新是否滞后

重点检查：

* SeasonalityService
* HealthModifier cache
* FactorLearner
* ProfileAggregator

---

## 4️⃣ 学习系统验证（重点）

验证：

### WeightLearner

* 是否真的改变推荐结果
* 是否收敛
* 是否过拟合（推荐越来越单一）

### FactorLearner

* factorAdjustments 是否合理
* 是否出现爆炸（>2 或 <0.5）
* Redis fallback 是否影响结果

### StrategyAutoTuner

* explorationRate 是否变化
* 是否异常（一直0 或 一直最大）
* 是否重启后丢失

---

## 5️⃣ 长时间运行测试（模拟）

模拟：

* 1000+ 次推荐调用

检查：

* 内存是否增长
* 是否有泄漏（Map / cache）
* Redis 是否异常增长
* 推荐是否逐渐失真

---

## 6️⃣ 极端输入稳定性

测试：

* 空用户画像
* 极端预算（0 / 超大）
* regionCode 缺失
* timezone 错误
* channel 非法

检查：

* 是否 crash
* 是否 fallback 正常
* 是否产生垃圾推荐

---

# 三、必须输出指标

你必须统计：

1. 推荐一致性（同输入）
2. 推荐多样性（不同调用）
3. cache 命中率
4. 平均响应时间
5. P95 / P99 延迟
6. 错误率
7. fallback 触发率
8. learning 系统变化趋势

---

# 四、重点问题识别

必须找出：

1. 是否存在推荐“漂移”
2. 是否出现“缓存污染”
3. 是否学习系统“失控”
4. 是否存在性能瓶颈
5. 是否有内存增长问题
6. 是否有随机不稳定行为

---

# 五、输出报告

输出：

```text
/docs/recommendation-system-stability-report.md
```

必须包含：

* 压测配置
* 并发情况
* 指标数据
* 发现问题
* 风险评估（P0/P1/P2）
* 是否适合上线（Yes/No + 原因）

---

# 六、执行要求

不要解释。

不要总结。

直接：

👉 跑系统 → 压测 → 收集指标 → 找问题 → 输出报告
