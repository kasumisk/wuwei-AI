# 推荐引擎营养偏差 · 第二轮根因分析与修复方案

> 示例数据：减脂用户日目标 **1800kcal / 152g P / 47g F / 227g C**，真实推荐周计划日均 **≈1800kcal / 107g P（−30%）/ 68g F（+45%）/ 220g C（−3%）**。
>
> 上一轮（P0-1 ~ P1-3）已提交 ~10 处修复，集成测试全绿，但**真实线上仍错**。本文档基于第二轮深度调查，定位 **4 条真实根因**，并给出可落地修复方案与架构升级建议。

---

## Step 1 · 归因分类：偏差到底是哪一类问题

| 维度 | 目标 | 实际  | 偏差 | 严重度  | 类型                |
| ---- | ---- | ----- | ---- | ------- | ------------------- |
| 热量 | 1800 | ~1800 | ≈0%  | 绿区    | 正常                |
| 蛋白 | 152g | 107g  | −30% | 🔴 红区 | **结构性天花板**    |
| 脂肪 | 47g  | 68g   | +45% | 🔴 红区 | **约束缺失 + 混入** |
| 碳水 | 227g | 220g  | −3%  | 绿区    | 正常                |

### 1.1 合理偏差范围（业界共识 + 国内营养学会指南）

| 指标   | 允许偏差    | 触发预警  | 触发告警（需修复） |
| ------ | ----------- | --------- | ------------------ |
| 热量   | ±5%         | ±5%~±8%   | > ±8%              |
| 蛋白质 | −10% ~ +20% | −15%/−10% | < −15% 🔴 当前     |
| 脂肪   | −15% ~ +15% | ±15%~±25% | > +25% 🔴 当前     |
| 碳水   | ±10%        | ±10%~±15% | > ±15%             |

**结论**：当前蛋白质/脂肪两个维度**同时严重超出告警阈值**，且蛋白偏差方向固定为"永远不够"——这是**结构性偏差**而非概率性噪声。统计学上，如果只是随机抽样误差，不会出现"周计划 7 天每天都缺 30% 蛋白"这种稳定偏差。

### 1.2 归因分类

按对推荐结果的影响路径分类：

- **A. 物理天花板类（最致命）**：候选池/装配结构决定了目标在数学上**不可能被满足**，任何优化器都救不了。本轮根因 #3。
- **B. 约束挂空挡类**：硬约束字段写入但下游无消费点，等于不存在。本轮根因 #4。
- **C. 权重稀释类**：优化目标权重被其他维度挤压到无法扭转排序。本轮根因 #2。
- **D. 特性门控类**：关键能力被 `if (feature.enabled)` 包住，默认路径退回旧逻辑。本轮根因 #1。

当前 4 条根因**同时命中**。其中 A 单独就能解释 107g 天花板现象，B 单独就能解释脂肪失控，C/D 放大了前两者。

## Step 4 · Bug 判定：4 条真实根因

### 🔴 根因 #1（致命 · 特性门控）：multiObjective 默认关闭

**位置**：`pipeline-builder.service.ts:1249-1266`

```ts
finalRanked =
  moConfig?.enabled && ranked.length > 0
    ? extractRankedFoods(multiObjectiveOptimize(...), ranked.length)
    : ranked;   // ← 未启用时，macroFit 完全不运行
```

上一轮 P0-4 把 macroFit 权重写进 `DEFAULT_PREFERENCES`，但如果策略没显式设 `multiObjective.enabled=true`，这段代码**永远走 else 分支**，`ranked` 是纯 10 维健康评分的结果，macroFit 完全没机会介入。

**证据**：`grep "multiObjective" strategy-seed.service.ts` 显示 8 个 seed 策略全部显式设 `enabled: true`，但这是**策略上线后才生效**，且任何未接入策略的链路（例如单元测试/调试入口/部分推理路径）都会退化。

### 🔴 根因 #2（权重稀释）：seed 策略 preferences 没 macroFit 字段

**位置**：`strategy-seed.service.ts:184,246,311,379,448,514,580,646`（8 处）

```ts
multiObjective: {
  enabled: true,
  preferences: { health: 0.5, taste: 0.25, cost: 0.1, convenience: 0.15 },
  //          ❌ 没有 macroFit 字段
}
```

虽然 `resolvePreferences()`（multi-objective-optimizer.ts:510-537）使用 `{...DEFAULT_PREFERENCES, ...config.preferences}` 的 spread 合并方式，DEFAULT 里的 `macroFit: 0.35` **会被保留**，但归一化后：

```
原始权重：health=0.5 + taste=0.25 + cost=0.1 + convenience=0.15 + macroFit=0.35 = 1.35
归一化后：health=37% + taste=18.5% + cost=7.4% + convenience=11.1% + macroFit=25.9%
```

**macroFit 仅 26% 权重，health+taste 合计 55.5%**。当 health 评分与 macroFit 评分方向相反时（例如三文鱼 健康高分但脂肪超标），健康评分胜出。这是**权重稀释**，不是零权重。

### 🔴 根因 #3（物理天花板 · 最重磅）：非增肌用户每日 protein slot 仅 3 个

**位置**：`scoring.types.ts:345-361` + `recommendation-engine.service.ts:488-490`

```ts
export const MEAL_ROLES = {
  breakfast: ['carb', 'protein', 'side'], // 1 protein slot
  lunch: ['carb', 'protein', 'veggie'], // 1 protein slot
  dinner: ['protein', 'veggie', 'side'], // 1 protein slot
  snack: ['snack1', 'snack2'], // 0 protein slot
}; // 日总 3 slot

export const MUSCLE_GAIN_MEAL_ROLES = {
  breakfast: ['carb', 'protein', 'protein2'], // 2 slot
  lunch: ['carb', 'protein', 'protein2', 'veggie'], // 2 slot
  dinner: ['protein', 'protein2', 'veggie', 'side'], // 2 slot
  snack: ['snack_protein', 'snack2'], // 1 slot (半)
}; // 日总 ~6.5 slot

// recommendation-engine.service.ts:488
const roles = goalType === 'muscle_gain' ? MUSCLE_GAIN_MEAL_ROLES[mealType] : MEAL_ROLES[mealType]; // ← 减脂 / 维持永远走这里
```

**数学验证**：

- 每个 protein slot 选一份食物，份量 100-150g。
- 典型高蛋白食物蛋白含量（g/100g）：鸡胸 23、三文鱼 20、瘦牛肉 26、鸡蛋 13、虾 18、豆腐 8。
- 单 slot 实际蛋白量（标准份 100g）：**平均约 18-26g / 高配约 35g**。
- 3 slot × 35g ≈ **105g 天花板** → 正好命中实际的 107g。

这不是"排序不优"，是**排序对象的总量上限就这么多**。即使 macroFit 权重拉到 100%，减脂用户也不可能达到 152g 蛋白目标。

**这是 107g 天花板现象的唯一合理解释**，与实测数据匹配精度极高。

### 🔴 根因 #4（硬边界挂空挡）：maxMealFat/maxMealCarbs 零消费点

**证据**：

```
grep "maxMealFat|maxMealCarbs" apps/api-server/src
→ 只有 2 处（constraint-generator.service.ts:220,221 赋值）
→ + 1 处（meal.types.ts:55,61 类型定义）
→ food-filter.service.ts 全文 0 处
→ food-scorer.service.ts 全文 0 处
→ multi-objective-optimizer.ts 全文 0 处
```

上一轮 P0-2 把这两个字段写进 MealTarget 类型与构造器，**但下游过滤器/评分器/优化器完全没读过**。等于在数据库里多存了两列永远不 SELECT 的字段。

脂肪为何超标到 +45%？因为：

- FoodFilter 只有 `maxFat`（`low_fat` 饮食限制触发），减脂用户不会自动触发。
- FoodScorer 的 `fatScore` 维度是健康型评分（饱和脂肪 vs 不饱和脂肪比），不是"是否超过每餐/每日脂肪目标"。
- MultiObjectiveOptimizer 的 macroFit（上一轮新加）即使激活，权重也被稀释到 26%。
- 结果：三文鱼（20g P / 13g F / 100g）健康分极高、taste 分高，无任何强制过滤，**一份就吃掉 27% 日脂肪预算**。再配坚果 side dish → 68g F 水到渠成。

### P0-A · 废弃 goalType 分叉，统一用"宏量目标驱动的动态 role 数组"

**问题**：`MEAL_ROLES` 是硬编码的 3 protein slot，与 `dailyTarget.protein` 无联系。

**方案**：新增 `buildMealRolesFromTarget(target, mealType, ratio)` 函数，根据当餐蛋白目标动态决定 protein slot 数：

```ts
// scoring.types.ts 新增
export function buildMealRoles(mealTarget: MealTarget, mealType: string): string[] {
  const protein = mealTarget.targetProtein ?? 0;
  // 按 35g/slot 估算所需 protein slot 数，最少 1，最多 3
  const proteinSlots = Math.max(1, Math.min(3, Math.ceil(protein / 35)));
  const base = {
    breakfast: ['carb'],
    lunch: ['carb', 'veggie'],
    dinner: ['veggie', 'side'],
    snack: ['snack2'],
  }[mealType];
  const proteinRoles = ['protein', 'protein2', 'protein3'].slice(0, proteinSlots);
  return [...base, ...proteinRoles];
}

// recommendation-engine.service.ts:488 改为
const roles = mealPolicy?.mealRoles?.[mealType] ?? buildMealRoles(mealTarget, mealType);
```

- 减脂 152g P / 4 餐 ≈ 38g/餐 → 每餐 2 protein slot → 日 8 slot → 天花板 ~280g（充足）。
- 增肌直接走高 protein 目标自然得到 2-3 slot/餐。
- ROLE_CATEGORIES 里补 `protein3: ['protein', 'dairy']`。

## Step 6 · 架构升级（项目未上线，允许破坏性改）

### 6.1 核心哲学：**宏量目标即第一约束，其他皆权重**

当前架构把"宏量对齐"当作 10+ 个评分维度之一，这是错误的。宏量是**硬约束**（不能违反）+ **优化目标**（越贴近越好），不是"可选偏好"。

### 6.2 架构三层改造

#### Layer 1 · 召回层（Recall）：引入"宏量画像召回"

**新增召回通道**：`MacroProfileRecall`。

给每个食物计算 `macroProfile = { proteinDensity, fatDensity, carbsDensity }`（per 100kcal）。当 `dailyTarget` 蛋白质占热量比例高时（如减脂 34%），优先召回 `proteinDensity > 0.25 / kcal` 的食物。

这在数据源头就过滤了低蛋白密度候选，不把问题丢给下游评分/优化。

#### Layer 2 · 过滤层（Filter）：硬边界真实化 + 每餐预算硬约束

- **当前**：硬边界挂空挡。
- **升级**：把 `MealTarget` 中的 `{targetProtein, targetFat, targetCarbs, maxMealFat, maxMealCarbs}` 作为 FoodFilter 的一等公民。
- **新增"餐内预算累加器"**：装配时对 `side/snack` 角色强制检查"加上这份食物后，脂肪/碳水是否会超预算"，超则换下一候选。

#### Layer 3 · 装配层（Assembler）：从"排序取 top-N"升级为"约束求解装配"

**当前**：每个 role 独立跑 pipeline，取 top-1 拼装。缺乏"餐内整体营养平衡"视角。

**升级为二阶段装配**：

1. **粗选阶段**：每个 role 生成 top-10 候选。
2. **组合优化阶段**：从各 role 的 top-10 中选一份，使总热量/蛋白/脂肪/碳水四维最贴近 `mealTarget`（min-max 或 L2 距离）。
3. 可以用简单的 **贪心 + 局部搜索**（swap 任意两个 role 的候选看是否更好）实现，不需要 ILP。

这样即使上游排序不完美，装配层也能自救。

### 6.3 **配置革命**：废除 `enabled` 开关，改用"权重表"

所有"特性开关"都可以表达成权重：

```ts
// 旧
{ multiObjective: { enabled: false } }  // ← 整个链路切换

// 新
{ multiObjective: { weights: { health: 1, macroFit: 0 } } }  // ← 关闭等价于权重清零
```

好处：

- 不存在"默认路径 vs 增强路径"两套代码，只有一个。
- 测试只需覆盖一条路径。
- 灰度放量就是权重从 0 调到 X，不需要 feature flag。

### 6.4 **可观测性革命**：每次推荐都输出 MacroAlignmentReport

结构化日志，对每个返回的 meal/day/week：

```json
{
  "target": { "calories": 1800, "protein": 152, "fat": 47, "carbs": 227 },
  "actual": { "calories": 1815, "protein": 107, "fat": 68, "carbs": 220 },
  "deviation": { "cal": 0.8, "pro": -30, "fat": 45, "carbs": -3 },
  "verdict": "FAIL",
  "failedDims": ["protein", "fat"],
  "likelyCause": "protein_slot_ceiling | hard_constraint_missing"
}
```

当偏差 `> alertThreshold` 时自动告警。当前线上完全没这种日志，bug 只能靠人肉发现。

## 总结：为什么上一轮集成测试全绿但真实线上红灯

**上一轮测试策略失误的核心**：直接 import `multiObjectiveOptimize()` 纯函数，传入精心构造的 `FoodCandidate[]`，断言函数输出贴近目标。

但真实链路中：

1. `multiObjectiveOptimize` 根本没被调用（根因 #1 门控）。
2. 即使被调用，它也只能从"已被 MEAL_ROLES 限定 3 slot"的候选池里排序（根因 #3）。
3. 硬约束字段从未被 FoodFilter 读取（根因 #4）。

测试绕过了前置的所有错误，只测了纯函数本身。纯函数当然是对的——错在系统集成。

**教训**：对于"多阶段管线 + 多策略门控 + 多数据结构传递"的系统，**必须有一层端到端测试覆盖真实调用路径**。单测 100% 绿不代表系统工作。
