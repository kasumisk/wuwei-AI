# 推荐引擎营养目标偏差分析（Daily Status vs Recommendation 错位根因）

> 用户案例：Daily Status 目标 = { protein 144g, fat 37g, carbs 148g }；周计划日均实际 = { protein 91g（−37%）, fat 64g（+73%）, carbs 147g（≈0%）}。
> 以下分析严格基于仓库现状（非猜测），所有引用均带文件:行号。

**误差注入点（按权重从大到小）：**

1. **P0 — `recommendMeal` 签名截断**：`dailyTarget: { calories: number; protein: number }`，fat/carbs 目标被编译期丢弃，下游所有硬/软约束均无法看到脂肪与碳水的目标值。
2. **P0 — `MACRO_RANGES.fat_loss.fat=[0.20, 0.35]`**：把"35% 脂肪供能比"奖励为满分，与真实目标（22%）**方向性冲突**。案例数据正好落在区间中段（fat 64g × 9 / 1528 ≈ 37.6%），评分器认为"接近最优"。
3. **P1 — `ConstraintGenerator` 没有 maxFat 通用规则**：仅在 `low_fat` 饮食限制触发时才开，`fat_loss` 目标本身不触发。
4. **P1 — GlobalOptimizer 偏差权重偏低且受 minScoreRatio=0.85 压制**：fat 权重 0.12 是所有维度中最低的，同时只能替换池内候选、迭代 ≤12 轮，不足以把 +73% 偏差拉回。
5. **P2 — 两套蛋白质公式并存**：`calculateDailyGoals` 用 2.0 g/kg（fat_loss），`NutritionTargetService.calcProtein` 用 1.2 g/kg，微量营养评分与宏量目标基于两个不同的 protein 数值做判断。

## Step 3 ｜ 一致性检查（逐项验证）

### 3.1 推荐引擎是否以 Daily Status 为硬约束？

**否（仅部分）。** 只把 `calories` 做成 `maxCalories` 软上限，`protein` 做成**单食物** `minProtein` 下限（`food-filter.service.ts:108-111`）。fat、carbs 无任何硬约束通道。

### 3.2 推荐目标是"达成营养目标"还是"偏好 / 可执行性"？

**偏向后者。** `multi-objective-optimizer.ts:33-38` 默认偏好权重 `{health:0.4, taste:0.3, cost:0.15, convenience:0.15}`，其中 "health" 来自 `sf.score / maxScore` 的抽象分数（`multi-objective-optimizer.ts:235-238`），**不是到目标的距离**。也就是说：即使每个食物脂肪爆表但 health 分数高（因 NRF 微量营养密度好），依旧会被 Pareto 排上。

### 3.3 冲突项

| 冲突                          | 是否存在            | 证据                                                                                                     |
| ----------------------------- | ------------------- | -------------------------------------------------------------------------------------------------------- |
| 用户偏好 vs 营养目标          | ✅ 存在             | `multi-objective`：taste 权重 0.3，会与 health 并列 Pareto，推荐不会为对齐宏量而牺牲口味                 |
| 可执行性 vs 精确营养控制      | ✅ 存在             | `convenience/cost` 合计 0.3 权重；`GlobalOptimizer.minScoreRatio=0.85` 宁可偏差不变也不让替换降 15% 以上 |
| Daily Status 数值 vs 评分区间 | ✅ 存在（核心 Bug） | 目标要求 fat=22%，MACRO_RANGES 奖励 fat=35%；两个模块从未互相校验                                        |

### 3.4 周计划是否被"平均化策略"稀释？

**部分是。** `WeeklyPlanService.getWeeklyPlan` 并行调用 `dailyPlanService.generatePlanForDate`（`weekly-plan.service.ts:153-160`），**每天独立生成，没有周级再平衡**。如果单日平均有 +73% 脂肪偏差，周级会完整复制该偏差（并非更糟，但也不会被平均化修正）。`calcWeeklySummary` 只做只读汇总，不反馈回单日生成循环。

## Step 4 ｜ 是否是 Bug？

## ✅ 是 Bug — P0 级实现缺陷（非设计策略）

**根因模块：`RecommendationEngineService` 的接口契约 + `ConstraintGeneratorService` 的输出形态 + `MACRO_RANGES` 评分区间。**

三点不成立的理由，说明这**不是"允许偏差"的设计**：

1. 代码中存在 `GlobalOptimizer`，它本身就以"最小化 6 维偏差"为目标。既然下游已经在优化偏差，上游用 `{calories, protein}` 截断 dailyTarget 只能解释为**实现疏忽**，不是策略。
2. `calculateDailyGoals` 辛苦算出来的 fat/carbs 数字在最热路径（`recommendMeal / recommendMealFromPool / recommendByScenario`）中从未被消费，属于**死数据**。
3. `MACRO_RANGES.fat_loss.fat=[0.20, 0.35]` 与 `fatPercent.fat_loss=0.22` 之间的矛盾在一个 PR 里就能修复，典型配置飘移。

## Step 5 ｜ 修复方案（可直接落地，函数级定位）

### P0-1 ｜ 拓宽 dailyTarget 契约（必做）

- **文件**：`apps/api-server/src/modules/diet/app/services/recommendation-engine.service.ts`
- **改动**：把所有 `dailyTarget: { calories: number; protein: number }` 签名（L148、L376、L898，以及内部 `recommendMealFromPool`）全部改为 `MealTarget`（`recommendation.types.ts`，已有 `calories/protein/fat/carbs/fiber/glycemicLoad`）。
- **上游适配**：`daily-plan.service.ts:280, 504` 已经有 `goals.fat / goals.carbs`，直接传递即可（当前是人为截断）。

### P0-2 ｜ ConstraintGenerator 加入宏量硬边界

- **文件**：`constraint-generator.service.ts`
- **新增返回字段**：`maxFatDaily`, `minProteinDaily`, `carbsRange`。
- **示例实现**（伪代码）：
  ```ts
  const fatGap = dailyTarget.fat - consumed.fat;
  const carbsGap = dailyTarget.carbs - consumed.carbs;
  // 单餐硬上限 = 剩余脂肪预算 × (1 + 容错)
  const mealMaxFat = Math.max(0, fatGap) * (mealRatio + 0.05);
  const mealMinProtein = Math.max(0, proteinGap) * (mealRatio - 0.05);
  ```
- `FoodFilter` 已经支持 `maxFat`（`food-filter.service.ts:80-85`），只需让其在 `goal=fat_loss / muscle_gain` 时默认开启，并用**每餐脂肪预算 / 标准份量克数 × 100**换算成 per-100g 阈值。

### P0-3 ｜ 修正 MACRO_RANGES 与 Daily Goal 的一致性

- **文件**：`apps/api-server/src/modules/diet/app/recommendation/types/scoring.types.ts:245-253`
- **做法**：`MACRO_RANGES` 不再硬编码，改为 **从 dailyTarget 派生**：
  ```ts
  // 由 dailyTarget 计算理想供能比 ±5pp 为满分区间
  const proteinRatio = (dailyTarget.protein * 4) / dailyTarget.calories;
  const fatRatio = (dailyTarget.fat * 9) / dailyTarget.calories;
  const carbRatio = (dailyTarget.carbs * 4) / dailyTarget.calories;
  const macroRange = {
    protein: [proteinRatio - 0.05, proteinRatio + 0.05],
    fat: [Math.max(0.15, fatRatio - 0.05), fatRatio + 0.05],
    carb: [carbRatio - 0.05, carbRatio + 0.05],
  };
  ```
- 并且把 **protein 也作为区间维度**加入评分（目前蛋白质是分段函数，非区间函数，导致高于下限就无激励追平目标）。

### P0-4 ｜ MultiObjectiveOptimizer 新增 `macroFit` 维度

- **文件**：`multi-objective-optimizer.ts:33-38`
- **新默认偏好**：`{health:0.30, macroFit:0.25, taste:0.20, cost:0.125, convenience:0.125}`
- **新 dimension**：`computeMacroFitScore(food, mealTargetRemaining)`：以 1 − Σ|actual − target|/target 形式计算，满分 = 完美贴合剩余预算。

### P1-1 ｜ GlobalOptimizer 权重与约束调整

- **文件**：`global-optimizer.ts:29-36, 72`
- **改动**：
  - `DEVIATION_WEIGHTS.fat: 0.12 → 0.22`（与 protein 并列一线指标，因为脂肪偏差对健康影响不亚于蛋白质）。
  - `minScoreRatio: 0.85 → 0.70`（允许更激进的替换），同时 `maxIterations: 12 → 25`。
  - 新增"宏量偏差熔断"：当 protein 偏差 > 20% 或 fat 偏差 > 30% 时，强制触发跨餐高蛋白/低脂替换。

## Step 6 ｜ 架构级升级建议

### 6.1 引入 Nutrition Target Layer（强烈推荐）

把"营养目标"从 `DailyNutritionGoals / NutritionTargets` 两个分散结构升级为**单一领域对象 `NutritionTarget`**：

```ts
interface NutritionTarget {
  // 宏量（硬目标 ± 容错）
  macros: {
    calories: { target: number; tolerance: number }; // ±7%
    protein: { target: number; tolerance: number }; // ±10%，下限比上限严格
    fat: { target: number; tolerance: number }; // ±10%
    carbs: { target: number; tolerance: number }; // ±10%
  };
  // 微量（上/下限）
  micros: NutritionTargets; // 现有 micro 结构
  // 供能比（派生）
  energyDistribution: { proteinPct; fatPct; carbPct };
  // 元数据
  origin: 'profile' | 'coach_override' | 'experiment';
  version: string;
}
```

- **单一真理源（Single Source of Truth）**：所有下游（Recommendation / Analyze / Decision / Explain / Score）只能通过 `NutritionTargetProvider.get(userId, date)` 读取。
- **禁止截断传参**：废弃 `{calories, protein}` 这种偷懒结构。

### 6.2 Recommendation → Goal Alignment 中间模块（新增）

在 PipelineBuilder 与 ConstraintGenerator 之间插入 `GoalAlignmentPlanner`：

```
NutritionTarget → GoalAlignmentPlanner → MealBudget[breakfast, lunch, dinner, snack]
                        │
                        ├─ 根据已摄入（consumed）动态分配剩余预算
                        ├─ 处理弱时段惩罚 / 暴食风险收紧
                        └─ 输出带硬约束的 MealBudget（含 fat/carbs 上下限）
```

该模块单元测试目标：**对任意 profile + consumed 组合，输出的 4 餐 MealBudget 之和必须 = NutritionTarget.macros（± 5g 整数化误差）**。这是当前系统完全缺失的不变量。

### 6.3 闭环：Daily Score 反向影响推荐（强烈推荐）

当前 `nutrition-score.service.ts` 计算完分数后终止。建议：

- 当连续 3 天某宏量偏差 > 15% 时，自动 boost 下次推荐中该维度的软约束权重（feedback loop）。
- 接入已有的 `weight-learner.service.ts` / `factor-learner.service.ts`，让学习型权重把"用户 Daily Score 偏差"作为额外 training signal。

### 6.4 可观测性

- 在 `pipeline-builder.service.ts` 的 Trace 中新增 `macroDeviation`（target vs picked）的每阶段快照，便于 admin debug（与现有 `recommendation-debug.service.ts` 对齐）。

## 🔥 加分项 ｜ 容错区间设计

### 合理偏差范围

基于 ADA / USDA / 中国 DRIs 的临床营养实践：

| 指标   | 合理日偏差       | 合理周均偏差 | 案例实际 | 评估          |
| ------ | ---------------- | ------------ | -------- | ------------- |
| 热量   | ±7%              | ±5%          | ≈ +2%    | ✅ 正常       |
| 蛋白质 | ±10%（下限 −5%） | ±7%          | **−37%** | ❌ 严重异常   |
| 脂肪   | +15% / −20%      | ±10%         | **+73%** | ❌ 灾难级异常 |
| 碳水   | ±12%             | ±10%         | −0.7%    | ✅ 正常       |
| 纤维   | −10%（上不限）   | −5%          | -        | -             |

**结论：当前案例 50%+ 偏差远超任何临床或工程可接受阈值，属于系统性 Bug，不是策略容忍。**

### 容错区间设计模式

推荐使用 **三层阈值**（在 `NutritionTarget.macros.tolerance` 中显式建模）：

```
┌───────── Green Zone (±5%) ───────── ┐  无惩罚，满分
├───── Yellow Zone (5-15%) ──────────┤  软约束：评分打折，触发弱 boost
└───── Red Zone (>15%) ──────────────┘  硬约束：强制 GlobalOptimizer 熔断 + coach 提示
```

- **Green → 设计意图**：给予引擎策略自由度（口味/可执行性）。
- **Yellow → 学习信号**：喂入 feedback loop，下轮推荐收紧。
- **Red → 不可接受**：必须在返回前通过 GlobalOptimizer 跨餐替换或 portion 调整回到 Yellow。

当前系统**只有 Green Zone 的奖励**，缺 Yellow/Red Zone 的惩罚与熔断，这正是 50% 偏差能被返回给用户的深层原因。

## 附：为什么案例数据恰好呈现这个形态

> 已知：用户 fat_loss，weight≈72kg（由 144g protein / 2.0 反推）或 muscle_gain weight=65kg，calorieGoal=1500。
> 推荐输出：protein 91g（−37%）、fat 64g（+73%）、carbs 147g（≈0）。
> 总热量：91×4 + 64×9 + 147×4 = **1528 kcal**。

这组数据**完美命中以下三个"引擎视角的最优点"**：

1. **热量被命中**（1528 ≈ 1500 × 1.02）→ 引擎认为 calories 约束已满足。
2. **脂肪供能比 = 64×9 / 1528 ≈ 37.7%**，正好略超 `MACRO_RANGES.fat_loss.fat.upper=0.35`，是"边界最优"。
3. **蛋白质供能比 = 91×4 / 1528 ≈ 23.8%**，落在 `MACRO_RANGES` 未约束的"自由区"，加上单食物 `minProtein` 仅是餐级软下限（×0.5），不构成全天硬约束。

**反过来看 Daily Status 目标**（protein 38% / fat 22% / carbs 39%），要求推荐必须偏离"引擎默认最优区间"去迎合个性化配比，**但没有任何机制把这种偏离信号传到引擎**——于是引擎返回自己认为的最优解，与用户目标系统性错位 50%+。

这就是为什么修复必须是链路级（Step 5 的 P0-1 到 P0-3 必须一起做），任何单点修复（例如只调 `MACRO_RANGES`）都会被上游的 `dailyTarget` 截断再次稀释。
