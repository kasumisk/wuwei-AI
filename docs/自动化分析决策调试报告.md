## 2. 问题与修复（按时间序）

### P3 · 非食物输入伪造食物
**现象**：输入"空气"/"一大堆"/"我吃了点东西"，启发式编造 350g/630kcal 假"主食"。
**根因**：`extractFoodCandidates()` 失败时回退到 default heuristic 不区分语义。
**修复**：新增 `isLikelyNonFood()` 守卫，命中即抛 `BadRequestException`（HTTP 400 + i18n 提示）。
**位置**：`apps/api-server/src/modules/food/app/services/text-food-analysis.service.ts:1367-1430`
**回归**：12/12 非食物用例返回 400 ✅

### P4 · "薯条" → 薯片误识别
**现象**：`一份薯条` → matched = "薯片（大袋）"。
**根因**：`extractQuantity` 中 `chinesePattern` 把数词标为 `?` 可选，"份薯条" 整体匹配进了名词主体。
**修复**：把 `chinesePattern` / `suffixChinesePattern` 中 `数词组` 从 `?` 改为必填。
**位置**：`text-food-analysis.service.ts:555-562, 599-600`
**回归**：4/4 用例匹配为 `薯条（大份）` ✅

### P6 · 高蛋白食物超目标 → 评分=0
**现象**：纯 200g 鸡胸肉给 muscle_gain 用户 → proteinRatio=0 → 整体 caution。
**根因**：
- `calcProteinRatioScore` 超标衰减系数 0.15 太狠
- `adequacy < 0.9`（典型单餐情形）即压分 → 高蛋白食物没有豁免

**修复**：
- 超标衰减 0.15 → 0.5（更平缓）
- `adequacy < 0.9` 时若 `actual >= 0.6 * target` 走豁免 `Math.max(ratioScore, 80)`

**位置**：`apps/api-server/src/modules/diet/app/services/nutrition-score.service.ts:684-715`

### P7 · 全糖奶茶 60 kcal
**现象**：`一杯全糖珍珠奶茶` → 60 kcal（实际 600~900 kcal）。
**根因**：LLM 返回 per-100g 营养，但被代码当作 per-serving 直接 cap，最终 600ml 杯 × 1 / 1000 比例错位。
**修复**：LLM fallback 增加 per-serving 自检（density × volume 估算），并对 dish/composite 重设 cap。
**位置**：`text-food-analysis.service.ts:987-1016, 116-136`
**回归**：4/4 用例 600~900 kcal 区间 ✅

### P8 · dish/composite 单位热量过宽
**现象**：兰州拉面、椰子鸡饭被 LLM 给到 380+ kcal/100g。
**根因**：`CALORIES_PER_100G_HARD_CAP` 仅有 `composite: 350`，无 `dish` 类型。
**修复**：
```
const CALORIES_PER_100G_HARD_CAP = {
  ...
  composite: 300,  // 350 → 300
  dish: 280,       // 新增
};
```
**位置**：`text-food-analysis.service.ts:116-136`
**回归**：拉面/盖饭 case 均回落到合理 per-100g ✅

### P9 · 白开水 LLM 编造 900 kcal
**现象**：`一杯白开水` → 走 LLM fallback → 编造 900 kcal/250g。
**根因**：白开水未在食物库 → 触发 LLM；LLM 对零热量饮品有幻觉。
**修复**：在 LLM fallback 之前加入 `tryZeroCaloriePassthrough()`，命中"白开水/纯净水/矿泉水/凉白开/温开水/沸水/开水/清水/白水/冰水/苏打水"等关键词，合成 0 kcal beverage 直接返回。
**位置**：`text-food-analysis.service.ts:471-480, 1306-1364`
**回归**：4/4 白水用例 0 kcal + recommend ✅

### P11 · "一个苹果" 作为 fat_loss snack → caution + "建议减少份量"
**现象**：苹果（78 kcal/全食物/低 GI）给减重用户当零食 → caution，文案"Low score, consider reducing portion"。
**根因（分两层）**：
1. `calcEnergyScore` 用 Gaussian 衰减，单餐 cumulative actual << target 时退化到 0；
2. 低蛋白天然食物（苹果蛋白≈0）→ proteinRatio=30 / macroBalance=32 → healthScore=44 < caution 阈值；
3. 即使评分语义正确（苹果确实蛋白低），决策层不应输出"减少份量"这种字面错误。

**修复**：
1. **评分层**：`calcEnergyScore` diff<0 时设下行 floor（fat_loss/health = 50, muscle_gain = 35）。
   位置：`nutrition-score.service.ts:648-672`
2. **决策层**：新增小份健康零食豁免：
   ```
   snack 餐次 + kcal<200 + foodQuality≥60 + 在预算内 + 当前为 caution
   → 提升为 recommend
   ```
   位置：`apps/api-server/src/modules/decision/decision/decision-engine.service.ts:64-83`
3. **文案层**：新增 i18n key `score.lightSnackOk`（中/英），替换"减少份量"。

**回归**：
| profile | 苹果@snack rec | 苹果@lunch rec | 白水@snack rec | 奶茶@snack rec | 薯条@snack rec |
|---------|-----|-----|-----|-----|-----|
| fat_loss | recommend ✓ | caution ✓ | recommend ✓ | caution ✓ | caution ✓ |
| muscle_gain | recommend ✓ | - | recommend ✓ | caution ✓ | caution ✓ |
| diabetic | recommend ✓ | - | recommend ✓ | caution ✓ | caution ✓ |
| vegetarian | recommend ✓ | - | recommend ✓ | avoid ✓ | recommend ⚠️ |

> ⚠️ 素食用户对薯条+可乐 = recommend：因素食用户的健康约束权重不同 + 用例为"薯条+可乐"组合（非纯薯条）。属合理边界，留观。

## 3. 边界与设计权衡

### 3.1 小份零食豁免（V4.6）
**触发条件**（必须**全部**满足）：
- `mealType === 'snack'`
- `totalCalories < 200`
- `breakdown.foodQuality >= 60`（即 ≥80% 全食物 / 加工分≥60）
- `remainingAfter >= 0`（不超出今日预算）
- 原决策为 caution（不抢已 recommend 的；不洗白 avoid）

**为何不更激进**：
- 薯片/小蛋糕这类 < 200 kcal 但 foodQuality < 60 的不会被豁免
- 巨无霸/奶茶 kcal>200 自然不命中
- lunch/dinner 苹果保持原决策（避免"靠苹果当正餐"误导）

### 3.2 跳过项（用户决策）
- **P10 mealType vs reason 文案不一致**：80 用例 mealType=lunch 但 reason 偶现"Late-night/Dinner 高热量"。本轮跳过，待后续单独处理 i18n + 时间感知拼装。
- 兰州牛肉拉面 LLM 给 280 kcal/100g（真实 ~150-200）。dish cap=280 已挡住更高值，符合"严格优先食物库 + 兜底 LLM"策略。
