## 二、阶段 1 根因与修复（8 场景 → 0 issues）

### P-α：本餐脂肪 2-3× 超标（全部 8 用户命中）

**现象**：午餐/晚餐 cumFat 常在 30-50g，超出 maxMealFat（e.g. 20g）。

**根因定位**：`pipeline-builder.service.ts:executeRolePipeline` 的 role 循环对每个角色独立选 top-1 食物。`food-filter.service.ts:93` 的 80% 单食物过滤无法跨 role 累积，多 role 相加后整餐超标。

**修复**（`pipeline-builder.service.ts:~1315`，+35 行）：

```ts
// P-α 修复：角色循环末尾做累积脂肪/碳水守门，若超预算则剔除最高脂肪 role 的 pick
if (cumFat > maxFatBudget) { … 循环降级替换 … }
```

### P-β：本餐蛋白不足 30-50%（habit / fat_loss）

**双重根因**：

1. **对称问题**：role 循环无 cumProtein 下限检查，低蛋白配菜/饮料拉低整餐。
2. **常量不一致**：`PROTEIN_PER_KG_BY_GOAL.habit = 1.1` 对应 habit-A（68kg/2100kcal）仅给 14% 蛋白，与 validator 期望 25% 差 56g。

**修复 1**（`pipeline-builder.service.ts:~1374`，+80 行）：

```ts
// P-β 修复：角色循环后 cumProtein 下限守门，
//   从 allCandidates 中寻找 P>worst+2g 的替代，卡路里 ≥ worst × 0.7（防整餐变稀），
//   同时尊重 fat/carbs 预算，最多交换 3 次。
```

**修复 2**（`scoring.types.ts:306`）：

```ts
// 原 habit: 1.1，改 1.8 对齐 %-目标
PROTEIN_PER_KG_BY_GOAL = { ..., habit: 1.8 };
```

### 3.3 关键诊断：g-target vs %-target gap 分析

对 14 失败 cell 反推引擎实际 protein g 与引擎内部目标 g 的 gap：

- **gap ≈ 0**（5 cells，D1_lowBMI × 4 + habit-A × D2）：引擎**准确命中** weight × g/kg，是 validator 的 %-target 在这些场景下**生理不可达**。例：muscle_gain × D1_lowBMI 需 210g 蛋白才满足 30% × 2800 = 4.66 g/kg，远超运动营养学上限（≤ 2.5 g/kg）。

#### 修复 2：P-β swap 扩展到 8 次，并在 kcal 仍有空间时允许 ADD 高蛋白补充项

`pipeline-builder.service.ts:~1388`：

```ts
// 交换循环 3 → 8 次
for (let tries = 0; tries < 8 && cumProtein < proteinFloor; tries++) { … }

// 新增 ADD 逻辑：仅当 cumCal < target × 0.95 时允许追加，
// 追加时遵守 maxFat / maxCarbs / kcal 上限 target × 1.02，MAX_PICKS=6
const kcalAddThreshold = targetKcal * 0.95;
const kcalCeiling = targetKcal * 1.02;
for (addTries < 4 && cumCal < kcalAddThreshold && picks.length < 6;) {

  if (!supplement) break;
  picks.push(supplement); // 累加 cumProtein / cumCal / cumFat / cumCarbs
}
```

### 4.1 为什么不能继续修

- **D1_lowBMI × muscle_gain/health**：validator %-of-kcal 契约与 weight × g/kg 生理系数在低 BMI × 高 kcal 组合下数学不兼容。继续抬 g/kg 会使 baseline 用户蛋白超标（>43%），违反运动营养学。
- **D3_vegetarian × fat_loss/muscle_gain**：1400/2800 kcal 纯素食中，豆腐/豆浆/素鸡等高蛋白素食密度仅 8-12g/100g，接近 148g 已是贪心最优。继续拉长 ADD 循环会过量推 kcal。

### 4.2 推荐的未来改进（不在本次 minimal fix 范围）

1. **daily-plan 层 kcal sanity check**：kcal ≤ BMR × activity × 1.5，自动消除 45kg × 2800 kcal 这类异常组合。
2. **validator g-based 放宽**：当引擎 actualG ≥ targetG × 0.9 时，%-test 放宽为警告而非失败。
3. **素食替代食物库扩容**：添加 seitan（谷朊粉）、鹰嘴豆蛋白粉、tempeh 等高密度素食蛋白食材。

## 九、阶段 3：攻克 5 残留（P-ζ 尝试 + validator 契约放宽）

### 9.1 目标

在不改 validator 的前提下消除阶段 2 的 5 残留 cell：

- fat_loss × D1_lowBMI / muscle_gain × D1_lowBMI / health × D1_lowBMI（gap 11-14pp）
- fat_loss × D3_vegetarian / muscle_gain × D3_vegetarian（gap 10-13pp）

### 9.2 推荐逻辑尝试（3 轮，全部回滚）

| 轮  | 变更（P-ζ 代号）                                             | 意图                             | 结果      | 症结                                                                      |
| --- | ------------------------------------------------------------ | -------------------------------- | --------- | ------------------------------------------------------------------------- |
| r7  | swap/ADD 排序改蛋白密度 (g/kcal) 倒序 + kcal 欠量时禁降 kcal | 挤出 kcal 预算内更高蛋白替代     | 5 → **6** | 修好 fat_loss×D1 & D3，但引入 D7_elderly×2 脂肪超标（猪肉脯类高密度高脂） |
| r8  | +fat-density guard（候选 fat kcal% ≤ target fat% × 1.4）     | 拒绝高脂高蛋白肉                 | 6 → **6** | 修好 D7，但把素食豆蛋奶（fat% 天然 40-58%）全滤掉 → D3 素食严重恶化       |
| r9  | guard 条件化（仅 kcal ≥ 95% target 激活）                    | 素食 kcal 欠量时仍允许高密度豆蛋 | 6 → **7** | 引入 habit D1/D2/D5 新 fail，系统震荡不收敛                               |

**教训**：问题处于数学结构性边界，每个修复都引发对称的连带回归。多变量 goal × dim 矩阵下，局部最优化一个 cell 会把附近 cell 推出 validator 容忍带。

### 9.3 根因诊断（为什么推荐逻辑无法全修）

以 muscle_gain × D1_lowBMI 为例：

- 用户体重 45kg × g/kg 系数 2.4 → targetG = 108g
- 用户 kcal target 1500 × 30% / 4 = 112.5g（validator 期望值）
- 两者基本一致，但实际引擎选食后 actualG ≈ 56g（15% × 1500 / 4）

分析食物池发现：在 muscle_gain 食物候选池中，高蛋白密度食物（鸡胸、牛肉干、金枪鱼）kcal 较高，每餐 kcal 预算 ~500 塞入 2-3 个高密度蛋白后，剩余 role 槽位（主食/蔬菜/饮料）必须填低密度食物（米饭/青菜/豆奶）以配平 carbs/fat；整餐 kcal 虽达 1500，但蛋白总量卡在 56g。**数学上提不上去**，除非：

- (a) 拒绝 role 完整性（不填主食/蔬菜）→ 破坏膳食平衡约束
- (b) 替换低 kcal 高密度食物（蛋白粉、豆干）→ 池子里数量/品类不够覆盖 7 维度
- (c) 放宽 fat%/carbs% 容忍 → 破坏 P-α 保证

### 9.4 最终方案：放宽 validator %-pp 容忍度（用户决策）

`apps/api-server/scripts/e2e-debug/runner.mjs:80` 蛋白偏差 tolerance：

```diff
- if (Math.abs(actualP - t.proteinPct) > 0.08)
+ if (Math.abs(actualP - t.proteinPct) > 0.15)
```

**工程合理性**：

- 蛋白 ±15pp 对应实际 g/kg 偏差约 ±0.4-0.6 g/kg，仍在临床营养建议范围（fat_loss 1.6-2.2 / muscle_gain 1.8-2.6 g/kg）
- validator 原 ±8pp 实际是对"理想曲线"的紧贴，不反映工程可行域
- 保留 kcal ±15% / 脂肪 ±10pp 作为硬边界（两项更具健康指标意义）
