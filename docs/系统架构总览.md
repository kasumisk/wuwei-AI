## 1. 系统组成

系统采用 **Monorepo** 架构（Turborepo + pnpm workspace），分为四个层次：

### 1.1 前端层

| 应用                  | 技术栈                                 | 端口 | 职责                                       |
| --------------------- | -------------------------------------- | ---- | ------------------------------------------ |
| **Web（C端）**        | Next.js 15 + TypeScript + Tailwind CSS | 3000 | 用户主应用，提供食物分析、推荐、订阅等功能 |
| **Admin（后台）**     | Vite 6 + React 19 + Ant Design         | 5173 | 管理后台，数据看板、用户管理、内容管理     |
| **MiniApp（小程序）** | 微信小程序                             | -    | 微信生态入口                               |

**前端职责：**

- 用户交互界面渲染
- 本地状态管理（Zustand + React Query）
- API 请求封装与错误处理
- 路由与页面导航（App Router / React Router）
- 订阅状态管理与付费弹窗
- i18n 国际化

### 1.2 后端层

| 组件           | 技术栈                 | 端口 | 职责                            |
| -------------- | ---------------------- | ---- | ------------------------------- |
| **API Server** | NestJS 10 + TypeScript | 4000 | 核心业务逻辑、API网关、认证授权 |

**后端职责：**

- RESTful API 服务（~100+ 端点）
- 多策略认证（JWT、匿名、邮箱、手机、微信）
- 订阅与支付处理（Apple IAP + 微信支付）
- AI 能力网关（多模型路由与降级）
- 异步任务处理（BullMQ）
- 事件驱动的模块间通信（EventEmitter2）

### 1.3 数据层

| 组件           | 技术             | 职责                                       |
| -------------- | ---------------- | ------------------------------------------ |
| **PostgreSQL** | TypeORM + Prisma | 主数据库，55个实体，支持 pgvector 向量搜索 |
| **Redis**      | ioredis          | 二级缓存（L1 内存 + L2 Redis）、会话、限流 |
| **BullMQ**     | 基于 Redis       | 异步任务队列（食物分析、通知推送、预计算） |
| **R2**         | Cloudflare R2    | 文件存储（图片上传）                       |

**数据层职责：**

- 结构化数据持久化
- 高性能缓存与会话管理
- 异步任务调度与执行
- 文件/图片存储

### 1.4 AI 能力层

| 组件                   | 模型                                  | 用途                                  |
| ---------------------- | ------------------------------------- | ------------------------------------- |
| **OpenRouter**         | ERNIE 4.5 8K                          | 食物文本分析（轻量）                  |
| **OpenRouter**         | ERNIE 4.5 VL 28B                      | 食物图片分析（视觉）                  |
| **OpenRouter**         | DeepSeek Chat V3                      | AI 教练对话                           |
| **Gateway 多模型路由** | OpenAI / DeepSeek / Qwen / OpenRouter | 通用文本/图片生成，自动降级           |
| **LangChain**          | ChatOpenAI（动态路由）                | RAG 管道，文档上传与查询              |
| **Food Pipeline**      | 通过 Gateway                          | 食物数据 ETL：AI 标注、翻译、图片识别 |

**AI 层职责：**

- 食物识别与营养分析
- 个性化对话与教练服务
- 多模型智能路由与降级
- 食物数据自动化处理

## 2. 核心模块

### 2.2 核心模块详解

#### 用户系统（AuthModule + UserModule）

**职责：** 用户全生命周期管理

| 子系统       | 说明                                                              |
| ------------ | ----------------------------------------------------------------- |
| 多策略认证   | 匿名登录 → 邮箱/手机/微信注册 → 匿名升级                          |
| 三层用户画像 | 声明画像（Onboarding）+ 行为画像（自动计算）+ 推断画像（ML 细分） |
| Onboarding   | 4步引导：基础信息 → 饮食偏好 → 健康目标 → 过敏原                  |
| 画像缓存     | Redis 缓存 + CronJob 定时刷新                                     |

**关键实体：** AppUser, UserProfile, UserBehaviorProfile, UserInferredProfile, ProfileSnapshot

---

#### 食物分析系统（FoodModule）

**职责：** AI 驱动的食物识别与营养分析

| 功能     | API                                  | 说明                          |
| -------- | ------------------------------------ | ----------------------------- |
| 文本分析 | `POST /api/app/food/analyze-text`    | 输入食物名称，AI 返回营养数据 |
| 图片分析 | `POST /api/app/food/analyze`         | 上传图片，异步 AI 视觉识别    |
| 分析历史 | `GET /api/app/food/analysis/history` | 分层数据返回（订阅等级门控）  |
| 食物库   | `GET /api/foods/search`              | 全文搜索 + 分类浏览           |

**技术要点：**

- 文本分析：同步调用 ERNIE 4.5 8K，Redis 缓存结果
- 图片分析：异步 BullMQ 任务 → ERNIE 4.5 VL 28B 视觉模型 → 轮询结果
- 数据来源：USDA + OpenFoodFacts + 中国食物成分表（Pipeline ETL）

**关键实体：** FoodLibrary, FoodAnalysisRecord, FoodCandidate, AnalysisFoodLink

---

#### 推荐系统（DietModule - 推荐引擎）

**职责：** 个性化膳食推荐

系统最大模块（1433行多阶段推荐管道），包含5个核心组件：

| 组件                     | 职责                                   |
| ------------------------ | -------------------------------------- |
| **ConstraintGenerator**  | 根据用户画像生成营养约束               |
| **FoodFilter**           | 过滤不适合的食物（过敏原、限制等）     |
| **FoodScorer**           | 多维度评分（营养匹配、偏好、新鲜度等） |
| **MealAssembler**        | 组装完整膳食方案                       |
| **HealthModifierEngine** | 根据健康目标微调推荐                   |

**API 端点：**

- `GET /meal-suggestion` - 下一餐推荐
- `GET /daily-plan` - 今日膳食计划（懒生成）
- `GET /weekly-plan` - 周计划
- `POST /explain-why-not` - 解释为何不推荐某食物
- `GET /substitutes` - 食物替代建议

**关键实体：** DailyPlan, PrecomputedRecommendation, RecommendationFeedback, ABExperiment

---

#### 用户画像系统（UserModule - Profile 子系统）

**职责：** 构建多维用户画像，驱动推荐引擎

三层画像架构：

| 层次         | 数据来源     | 更新频率     | 内容                                     |
| ------------ | ------------ | ------------ | ---------------------------------------- |
| **声明画像** | 用户主动填写 | 手动         | 年龄、性别、身高、体重、健康目标、过敏原 |
| **行为画像** | 系统自动计算 | 每次用餐记录 | 饮食频率、偏好品类、时段习惯、营养趋势   |
| **推断画像** | ML 模型推断  | CronJob 定期 | 用户细分、流失风险、偏好预测             |

**场景感知：** 系统可检测当前用餐场景（早餐/午餐/晚餐/加餐），自动调整推荐策略。

**关键实体：** UserProfile, UserBehaviorProfile, UserInferredProfile, ProfileSnapshot, ProfileChangeLog

---

#### 订阅系统（SubscriptionModule）

**职责：** 付费变现与功能门控

| 组件     | 说明                                                       |
| -------- | ---------------------------------------------------------- |
| 订阅计划 | Free / Pro / Premium 三级                                  |
| 支付渠道 | Apple IAP（StoreKit 2 + S2S）+ 微信支付（JSAPI + Webhook） |
| 配额系统 | 功能级使用限制（UsageQuota）                               |
| 付费墙   | SubscriptionGuard 门控 + 触发日志                          |
| 结果分层 | 按订阅等级裁剪返回字段                                     |

**API 端点：**

- `POST /apple/verify` - Apple 收据验证
- `POST /apple/webhook` - Apple S2S 通知
- `POST /wechat/create-order` - 创建微信支付订单
- `POST /wechat/webhook` - 微信支付回调

**关键实体：** SubscriptionPlan, Subscription, PaymentRecord, UsageQuota, SubscriptionTriggerLog

---

#### 数据存储

| 存储类型       | 技术                  | 用途                                      | 实体数             |
| -------------- | --------------------- | ----------------------------------------- | ------------------ |
| **关系数据库** | PostgreSQL + TypeORM  | 业务数据持久化                            | 55                 |
| **向量数据库** | PostgreSQL + pgvector | 食物语义搜索（96维嵌入）                  | 内嵌于 FoodLibrary |
| **缓存**       | Redis（二级）         | L1 内存热数据 + L2 Redis 分布式缓存       | -                  |
| **队列**       | BullMQ（基于 Redis）  | FOOD_ANALYSIS / NOTIFICATION / PRECOMPUTE | 3 队列             |
| **文件存储**   | Cloudflare R2         | 图片上传                                  | -                  |

### 2.3 模块关系图

```
用户系统 ←──────→ 食物分析系统
  │                    │
  │    ┌───────────────┤
  │    │               │
  ▼    ▼               ▼
用户画像 ──────→ 推荐系统
  │                    │
  │                    │
  ▼                    ▼
订阅系统 ←──── 功能门控（SubscriptionGuard）
  │
  ▼
数据存储层（PostgreSQL / Redis / R2）
```

**模块间通信方式：**

- **直接依赖注入（DI）：** 模块间通过 NestJS 的 DI 容器直接调用
- **事件驱动（EventEmitter2）：** 解耦跨模块通信，如 `user.meal.recorded` → 行为画像更新

## 4. 数据流说明

### 4.1 数据流动总览

```
用户操作
   │
   ▼
前端应用（Next.js / 微信小程序）
   │ HTTP/HTTPS
   ▼
NestJS API Server
   │
   ├─→ Guards（认证/授权/限流）
   │
   ├─→ Controller（路由分发）
   │
   ├─→ Service（业务逻辑）
   │      │
   │      ├─→ Cache（Redis L1/L2）──→ 命中则直接返回
   │      │
   │      ├─→ AI Service（OpenRouter/Gateway）──→ AI 计算
   │      │
   │      ├─→ TypeORM Repository ──→ PostgreSQL（读/写）
   │      │
   │      └─→ BullMQ Queue ──→ 异步任务处理
   │
   ├─→ Interceptor（响应格式化）
   │
   ▼
返回标准响应 { success, code, message, data }
```

### 4.2 写数据库的场景

| 场景            | 写入实体                        | 触发条件          |
| --------------- | ------------------------------- | ----------------- |
| 用户注册/登录   | AppUser                         | 首次注册          |
| 完成 Onboarding | UserProfile                     | 每步保存          |
| 食物分析        | FoodAnalysisRecord              | 每次 AI 分析完成  |
| 记录饮食        | FoodRecord                      | 用户手动记录      |
| 生成日计划      | DailyPlan                       | 首次请求/再生成   |
| 推荐反馈        | RecommendationFeedback          | 用户提交反馈      |
| AI 决策         | AiDecisionLog                   | 每次决策判断      |
| 教练对话        | CoachConversation, CoachMessage | 每轮对话          |
| 支付完成        | Subscription, PaymentRecord     | 支付成功回调      |
| 配额更新        | UsageQuota                      | 功能调用/订阅变更 |
| 付费墙触发      | SubscriptionTriggerLog          | 用户碰到付费墙    |

### 4.3 使用缓存的场景

| 场景         | 缓存策略                  | TTL              |
| ------------ | ------------------------- | ---------------- |
| 食物分析结果 | 相同文本输入缓存          | 中等             |
| 用户画像数据 | 三层画像 Redis 缓存       | CronJob 定期刷新 |
| 推荐结果     | PrecomputedRecommendation | 按时段缓存       |
| 食物库搜索   | 热门搜索缓存              | 短时             |
| 订阅状态     | 用户订阅信息缓存          | 变更时失效       |

### 4.4 使用 AI 计算的场景

| 场景         | 模型             | 调用方式       | 耗时   |
| ------------ | ---------------- | -------------- | ------ |
| 食物文本分析 | ERNIE 4.5 8K     | 同步           | ~2-5s  |
| 食物图片分析 | ERNIE 4.5 VL 28B | 异步（BullMQ） | ~5-15s |
| AI 教练对话  | DeepSeek Chat V3 | SSE 流式       | 实时   |

## 8. 事件驱动架构

系统使用 `EventEmitter2` 实现解耦的模块间通信：

| 事件                             | 触发时机       | 响应动作                         |
| -------------------------------- | -------------- | -------------------------------- |
| `user.meal.recorded`             | 用户记录饮食   | 更新行为画像、连续打卡、成就检查 |
| `user.feedback.submitted`        | 提交推荐反馈   | 权重学习、A/B 数据记录           |
| `user.profile.updated`           | 画像变更       | 缓存失效、推荐重算               |
| `recommendation.generated`       | 生成推荐       | 预缓存、使用计数                 |
| `subscription.changed`           | 订阅状态变更   | 功能门控刷新                     |
| `food.analysis.submitted`        | 发起食物分析   | 分析生命周期跟踪                 |
| `food.analysis.completed`        | 分析完成       | 结果入库、通知用户               |
| `food.analysis.failed`           | 分析失败       | 错误日志、重试逻辑               |
| `food.analysis.saved_to_record`  | 分析保存为记录 | 关联分析与饮食记录               |
| `food.candidate.created`         | 新候选食物     | 食物数据治理                     |
| `subscription.paywall.triggered` | 触发付费墙     | 转化漏斗分析                     |

## 9. 可持续迭代结构

### 9.1 已有的扩展机制

| 机制               | 说明                                     |
| ------------------ | ---------------------------------------- |
| **模块化架构**     | NestJS Module 系统，新功能可独立模块接入 |
| **多模型 Gateway** | 新 AI 模型只需添加 Adapter               |
| **策略系统**       | StrategyModule 支持运行时策略切换        |
| **特性开关**       | FeatureFlagModule 支持功能灰度发布       |
| **A/B 实验**       | ABExperiment 支持推荐算法对比测试        |
| **事件驱动**       | 新模块可监听已有事件，零侵入扩展         |
| **食物 Pipeline**  | 可接入新数据源（USDA、OpenFoodFacts 等） |

### 9.2 建议的迭代方向

| 方向         | 说明                       | 优先级 |
| ------------ | -------------------------- | ------ |
| 社区功能     | 用户分享饮食、互动         | 中     |
| 智能提醒     | 基于画像的主动推送         | 高     |
| 运动整合     | 接入运动数据，热量平衡计算 | 中     |
| 多语言食物库 | 扩展国际食物数据           | 中     |
| 模型自训练   | 基于用户反馈微调推荐模型   | 低     |
